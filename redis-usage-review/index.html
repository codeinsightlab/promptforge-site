复制
## 1. 是否建议上线
- 结论：修改后可以
- 主要风险：删除 ZSet 任务和写入 pause context 不是原子操作，极端情况下可能出现任务已删除但暂停上下文未写入。
- 最小处理方向：保留 DB phase + roundToken 兜底，并尽量将删除任务与写入 pause context 封装为单一方法，必要时用 Lua 保证原子性。

## 2. Key 与数据结构评审
- Key 是否合理：基本合理，包含 orderId 和 taskType，能区分不同主时效任务。
- Value 是否足够：基本足够，remainingSeconds、reviewStartTime、roundToken 可以支持恢复和防旧轮次误恢复。
- 数据结构是否合适：Hash 合适，字段可读性好。
- TTL 是否合理：不完整。没有 TTL 可能导致异常流程下 pause context 长期残留。
- 是否存在 key 污染或无法清理：存在，需要设置兜底 TTL 或定期清理。

## 3. 并发与一致性风险
1. 风险：删除延迟任务成功，但写 pause context 失败
   - 触发场景：Redis 操作中途异常
   - 影响：审核拒绝后无法恢复主时效
   - 解决建议：封装原子操作；失败时抛异常并阻止进入审核态；必要时使用 Lua
2. 风险：旧 roundToken 的 pause context 被误恢复
   - 触发场景：用户多轮提交审核
   - 影响：旧任务重新入队，导致超时逻辑错误
   - 解决建议：恢复前必须校验 roundToken 与当前审核轮次一致
3. 风险：审核通过后 pause context 未清理
   - 触发场景：自动通过或人工通过分支遗漏删除
   - 影响：残留 key 干扰后续排查或误恢复
   - 解决建议：所有通过分支统一调用 clearPauseContext

## 4. 幂等与兜底评估
- 是否允许重复执行：不允许重复恢复旧任务
- 当前是否具备幂等：部分具备，依赖 roundToken
- DB 状态是否能兜底：可以，phase 和审核 round 是关键兜底
- 建议增加的状态校验：恢复前校验订单仍处于对应审核拒绝流程，且 roundToken 匹配

## 5. 原子性评估
- 是否需要 Lua：建议评估。删除 ZSet + 写 Hash 最好原子化。
- 是否需要分布式锁：不一定，优先使用状态校验和 roundToken。
- 是否存在 check-then-set 问题：存在，多步骤 Redis 操作之间可能被并发打断。
- 最小原子化方案：提供 pauseMainDelayTask(orderId, taskType, roundToken) 方法，内部完成 remove + put，并在失败时抛异常。

## 6. 监控与排查建议
- 需要记录的日志：orderId、taskType、remainingSeconds、roundToken、reviewStartTime、操作类型
- 需要监控的指标：pause context 数量、恢复失败次数、旧 roundToken 丢弃次数、ZSet 延迟任务数量
- 异常时如何排查：先查 DB phase 和审核 round，再查 pause context 和 ZSet 是否一致

## 7. 最小修改方案
- Key 调整：保留 order_delay_pause:{orderId}:{taskType}
- Value 调整：必须包含 roundToken、remainingSeconds、reviewStartTime
- TTL 调整：增加兜底 TTL，例如主时效最长周期 + 1 天
- 代码调整：封装暂停、恢复、清理三个方法；禁止业务代码分散操作 Redis
- DB 校验调整：恢复前强校验 phase 和 roundToken
- 风险等级：中
