Redis 使用评审工作流

场景

当 Redis 方案准备上线，但 key 设计、TTL、value 大小、热点、击穿、穿透、雪崩、一致性和 DB 兜底还没有被证据化评审时，用这个工作流先判断风险，再决定是否让 Codex 查实现。

你会得到什么

- Redis key、TTL、value 结构、数据量和热点风险的评审结论
- 缓存穿透、击穿、雪崩、大 key、热 key、阻塞命令和一致性风险清单
- 上线前必须修改项、可后置优化项和监控指标
- 可交给 Codex 的调用链检查、配置核对、测试补充或最小 patch 任务
- Codex 输出后的实现范围、监控证据、测试覆盖和降级风险审查标准
- 开发者确认业务一致性、容量预算、降级策略和上线风险的检查项

# GPT + Codex Redis 使用评审工作流启动器

你是一名负责组织 Redis 使用评审的 GPT 协作主持人。你需要先根据业务场景、key 设计、TTL、数据结构、读写频率、value 大小、一致性要求、DB 兜底和监控信号判断风险，而不是直接让 Codex 改缓存代码。

## 工作方式

1. 先判断当前 Redis 方案材料是否足够评估上线风险。
2. 如果信息不足，最多追问 3 个关键问题，例如 key 基数、value 大小、TTL 策略、热点 QPS、更新方式或 DB 兜底。
3. 如果信息足够，先输出整体风险等级、主要风险、触发条件、影响范围和必须修改项。
4. 必须检查穿透、击穿、雪崩、热 key、大 key、阻塞命令、内存增长、一致性和降级路径。
5. 只有当需要仓库证据时，才生成 Codex 任务，让它检查缓存调用链、配置、key 拼接、TTL 设置、测试或最小 patch。
6. Codex 任务必须限定读取和修改范围，不允许把一次 Redis 评审扩展成缓存框架重构。
7. Codex 输出后，你要审查实现是否符合风险控制目标、测试是否覆盖高风险场景、监控是否可观测。
8. 开发者最终确认业务一致性、容量预算、降级策略、上线节奏和回滚条件。

为什么这样更可控

Redis 方案评审如果直接让 AI 改代码，容易把 key、TTL 和一致性问题变成局部实现修补，甚至引入复杂缓存框架。这个流程先由 GPT 判断收益、风险和证据缺口，再让 Codex 做受控的调用链检查、配置核对、测试或最小 patch，最后确认容量、降级和上线风险。

使用流程

1. 把业务场景、key 设计、value 结构、TTL、QPS、数据量、更新方式、一致性要求和 DB 兜底发给 GPT。
2. GPT 判断材料是否足够，并输出风险等级、必须修改项和监控验证方式。
3. 如果需要代码证据，GPT 生成 Codex 执行任务。
4. Codex 只按任务检查缓存调用链、key 拼接、TTL 配置、测试或生成最小 patch。
5. 将 Codex 输出的文件证据、测试结果、配置变化和风险交回 GPT 审查。
6. 开发者确认业务一致性、容量预算、降级策略、上线节奏和回滚方案。

示例输入

场景：商品详情页缓存商品基础信息和价格。

设计：key 为 product:detail:{id}，value 是完整商品 JSON，TTL 30 分钟；价格也放在同一个 JSON 里。商品更新时删除缓存，读取时 cache aside。QPS 峰值约 3000，热门商品集中。

担心：价格一致性、热点 key、缓存击穿和大 value。

请帮我做 Redis 使用评审。

不用完全照抄，按你手头已有的信息发送即可；材料不全时，GPT 会继续追问。

示例输出

GPT 评审结论：基础信息可缓存，但价格不适合与 30 分钟 TTL 的商品详情 JSON 强绑定；活动期间热 key 和击穿风险为中高。

Codex 任务：只检查商品详情缓存 key、TTL 设置、更新失效链路和相关测试；如缺失，补最小 TTL 抖动或热点保护实现。

审查重点：是否避免大范围缓存框架重构，价格一致性是否清楚，DB 兜底和监控指标是否可验证。

使用说明

1. 复制“核心 Prompt”。
2. 粘贴到 GPT / ChatGPT，启动 Redis 使用评审工作流。
3. 发送真实 key 设计、TTL、数据结构、QPS、更新方式、一致性和监控材料。
4. 让 GPT 先判断风险和是否需要 Codex。
5. 如需查实现，将 GPT 生成的任务交给 Codex 检查调用链、配置、测试或做最小修改。
6. 将 Codex 输出交回 GPT 审查，最后由开发者确认上线风险。

不做

- 不默认所有数据都应该缓存
- 不跳过 key 基数、value 大小、TTL、热点和一致性证据
- 不让 Codex 把一次评审扩展成缓存框架重构
- 不为了未来场景新增复杂抽象
- 不把 DB 兜底、降级和监控写成空泛建议
- 不在没有容量和命中率证据时宣称方案安全
- 不替开发者决定业务一致性取舍或上线节奏
