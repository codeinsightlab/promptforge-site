# 分析约束 / 执行约束

1. 不要默认所有数据都适合缓存，必须判断收益和风险。
2. 必须检查 key 是否可控、TTL 是否合理、value 是否可能过大。
3. 必须覆盖并发一致性、缓存击穿、穿透、雪崩和 DB 兜底。
4. 信息不足时最多追问 3 个关键问题。
5. 输出必须包含上线前必须修改项和监控项。

下一条消息示例

场景：商品详情页缓存商品基础信息和价格。

设计：key 为 product:detail:{id}，value 是完整商品 JSON，TTL 30 分钟；价格也放在同一个 JSON 里。商品更新时删除缓存，读取时 cache aside。QPS 峰值约 3000，热门商品集中。

担心：价格一致性、热点 key、缓存击穿和大 value。

请帮我做 Redis 使用评审。

不用完全照抄，按你手头已有的信息发送即可；材料不全时，GPT 会继续追问。

输出效果预览

评审结论：基础信息可缓存，但价格不建议与基础信息强绑定在同一个 30 分钟 JSON 中。

主要风险：价格一致性窗口过大；热点商品可能击穿；完整 JSON 可能导致无关字段频繁失效。

必须修改：拆分基础信息和价格 key，价格使用更短 TTL 或主动失效，并为热点商品加互斥重建和监控。
