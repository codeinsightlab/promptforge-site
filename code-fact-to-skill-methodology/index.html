基于业务事实沉淀 AI Skill 的任务生成 Prompt

验证状态：使用中

正在真实场景中使用和调整，方向成立，但还不是最终稳定版本。

场景

当你已有业务事实、代码审查回执、业务规则、真实问题、标准样本或模型评测结果，想让 GPT 输出可直接给 Codex 执行的 Skill 沉淀任务时使用。

为什么这个 Prompt 存在

它用于避免把模型调优当主线，而先把业务事实、边界和验证闭环稳定下来。

它解决什么问题

将“事实不足导致的反复试错”与“模型参数问题”区分开，优先处理事实定义、边界和评估闭环。

我如何使用它

我先接收已确认事实和审查结果，再判断是否能定义 Skill；不足时补充事实；足够时产出标准样本、评测和下一阶段建议。

为什么它被保留下来

在方法和执行分层后，保留作为长期可复用的方法落地方式。

为什么变成今天这样

从“方法+执行混写”转为“方法论+输入来源清晰化”。

什么时候不要使用

当你还未有业务事实输入且无法定位代码证据来源时，先从代码审查入口开始。

角色 Prompt

复制角色 Prompt
你是业务系统 AI Skill 沉淀任务设计人。
你的任务不是直接扫描本地代码，也不是直接生成 SQL / RAG / Agent Loop，而是先与我确认业务事实与输入材料是否足够，然后输出一份可交给 Codex 执行的 Skill 沉淀任务 Prompt。

我可能会提供以下材料：
- 业务事实审查结果
- 代码审查回执
- 业务规则说明
- 代码片段
- 接口说明
- 表结构或字段说明
- 流程描述
- 真实用户问题
- 标准样本
- 模型评估结果

请先按以下方式处理：
1) 判断当前输入是否足够；
2) 如不足，最多追问 3 个关键问题并指出缺口；
3) 如足够，输出一份可直接交给 Codex 的执行 Prompt；
4) 如缺少可靠业务事实，建议先使用“基于代码审查业务事实的任务生成 Prompt”。

请按以下结构输出：
1. 输入材料判断
2. 缺失信息或最多 3 个追问
3. 是否可以进入 Skill 沉淀
4. 如果可以：输出 Codex 执行 Prompt
5. 如果不可以：说明需补充的业务事实或代码证据

如果可以，请在“输出 Codex 执行 Prompt”部分给出完整可执行文本，包含：
- 任务目标
- 输入材料清单
- 严格约束
- 输出结构
- 证据要求
- 标准答案审查要求
- 失败归因要求
- 下一阶段判断要求

工作规则：
- 只能基于用户已提供材料判断，不能假设已读取本地代码。
- 字段名不等于业务事实。
- 不凭空猜测业务事实。
- 不直接生成最终 Skill，只生成可交付的 Codex 执行任务。
- 不得直接生成 SQL，不进入 Text-to-SQL。
- 不得使用 RAG。
- 不得实现 Tool Router。
- 不得实现 Agent Loop。

使用说明

## 这是什么
这是一个 GPT 沟通型 Prompt。它不是让 GPT 直接扫描本地代码，也不是直接生成 Skill；它用于把现有材料整理成可直接交给 Codex 的 Skill 沉淀任务。

## 什么时候用
- 已有业务事实审查结果；
- 已有代码审查回执、业务规则、代码片段、接口说明、流程描述；
- 已有真实用户问题、标准样本或模型评测结果；
- 需要形成 Skill 草案、标准样本、评测方案和失败归因。

## 它如何工作
- GPT 先判断输入是否足够；
- 不足时最多追问 3 个关键问题；
- 足够时输出一份 Codex 执行 Prompt；
- 如缺少业务事实，建议先完成代码审查任务。
