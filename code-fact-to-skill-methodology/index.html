基于代码事实生成 AI Skill 的分层验证方法

验证状态：使用中

正在真实场景中使用和调整，方向成立，但还不是最终稳定版本。

场景

适用于已有业务系统里接入 AI 能力的场景，用于在模型调参前先把代码事实、事实边界与输出标准梳理清楚。

为什么这个 Prompt 存在

很多 AI 项目先把重点放在模型与提示词优化，导致在事实薄弱时反复试错。该方法用于先把业务事实边界稳定下来，再决定是否进入下游建模。

它解决什么问题

它将“模型问题”“协议问题”“工程问题”“标准答案问题”“事实定义问题”从结果层面拆开，让团队先对齐证据，再讨论优化方向。

我如何使用它

我先确认代码入口、核心对象、关键问题和输出协议，再走业务对象识别、事实抽取、事实目录与边界判断、Skill 草案与标准样本草案，最后用失败归因判断是否转入下一阶段。

为什么它被保留下来

在复用场景中它长期有效，能显著减少基于猜测的讨论，并提升团队对事实定义与验收边界的一致性。

为什么变成今天这样

已从“先改提示词、再追求分数”调整为“先建立事实边界，再决定是否进入模型优化”。

什么时候不要使用

当你只是想快速尝试一次性文本生成、或问题本身并非围绕代码事实建立 AI 能力时，不建议直接套用该方法。

角色 Prompt

复制角色 Prompt
## 这是什么

这是一套用于从代码事实出发建立 AI Skill 的方法论。它面向 Tech Lead 和 AI 工程判断者，目标是确认某一层 Skill 是否建立在真实的业务事实上，而不是在缺失证据的情况下直接优化模型。

## 适用场景

1. 已有业务系统希望接入 AI 问答，但没有稳定的事实边界。
2. 希望在 Text-to-SQL 之前建立一层可复用的语义事实层。
3. 需要从代码中抽取业务事实，生成事实目录与事实边界。
4. 需要输出 Skill 草案与标准样本。
5. 需要区分模型问题、工程问题、标准答案问题与事实定义问题。

## 核心原则

1. 不先问“如何让模型更准”，先问“代码里真实存在什么业务事实”。
2. 字段本身不等于业务事实。
3. Skill 是代码事实的语义封装，不是一般性的 Prompt 技巧。
4. 标准样本命中率高不等于业务正确。

## 方法流程

1. 代码审查
2. 业务对象识别
3. 业务动作 / 状态 / 规则抽取
4. 可查询或可判断事实抽象
5. 业务事实目录草案
6. 业务事实边界表
7. Skill 草案
8. 标准样本草案
9. 模型与执行链路评测
10. 标准答案审查
11. 失败归因
12. 收口或进入下一阶段

## 失败归因

- JSON 通过率低：排查响应协议与输出结构。
- 协议字段有效性低：检查字段命名、类型与约束。
- 规则失控高：排查 no-think / JSON-only 行为失效。
- 语义方向正确性低：确认事实语义边界是否清晰。
- 标准样本命中但人工不认同：标准答案可能有问题。
- 歧义点过多：可能是业务事实定义缺口，而非模型推理问题。
- SQL / 字段名泄漏：说明层级边界未分离干净。

你接下来可以这样输入

背景：目标层级为 {{TARGET_LAYER}}，已确认代码入口为 {{CODE_ENTRY_POINTS}}，并希望沉淀可复用的 AI Skill。
请先给出该层的业务对象识别、代码事实抽取与事实边界判断，再决定是否进入 Skill 草案与标准样本阶段。

GPT 会这样输出

先确认是否有足够代码证据定义业务对象与动作；
再输出业务事实目录草案与业务事实边界表；
仅在事实与协议边界清晰时推进 Skill 草案，后续以失败归因确认是否进入下一阶段。

使用说明

1. 先按方法论梳理当前问题是否属于事实定义、规范判断或模型优化前置阶段。
2. 先定义业务对象、规则、状态边界与输出协议。
3. 仅在事实边界稳定后，再启动配套的
   业务事实审查 Codex Prompt。
4. 对于模型、标准答案或工程争议，优先按失败归因闭环归类。
