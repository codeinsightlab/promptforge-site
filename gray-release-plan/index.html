灰度发布方案 Prompt

场景

用于设计功能上线前的灰度发布方案，重点关注灰度对象、放量策略、feature flag、数据兼容、监控指标、回滚条件和风险隔离。适用于新接口上线、支付链路切换、登录链路重构、新页面逐步开放和后端策略切换。

示例输出

复制
## 1. 是否建议灰度发布
- 结论：必须灰度
- 原因：该变更影响支付和会员开通，且涉及微信虚拟支付回调，不能直接全量。
- 最大风险：支付成功但会员未开通，或重复回调导致会员重复续期。

## 2. 灰度对象与范围
- 灰度对象：userId
- 第一批范围：内部测试账号 + 指定 20 个老用户
- 放量节奏：
  1. 内部账号验证 1 天
  2. 指定 20 个老用户验证 1 天
  3. 5% 用户观察 1 天
  4. 20% 用户观察 1 天
  5. 无异常后全量
- 白名单规则：白名单 userId 优先命中新链路；未命中继续走旧链路

## 3. Feature Flag 设计
- 开关名称：enable_virtual_vip_pay
- 命中规则：按 userId 白名单或百分比灰度命中
- 默认值：false
- 关闭方式：后台配置关闭后全部回到旧链路
- 是否需要后台配置：需要，至少支持开关和白名单

## 4. 发布步骤
1. 发布前准备：确认微信虚拟支付配置可用、旧会员支付链路保留、准备白名单账号
2. 数据库变更：新增 virtual_pay_order_no 字段，确认旧订单查询不受影响
3. 配置变更：增加 enable_virtual_vip_pay=false，配置微信虚拟支付参数
4. 后端发布：发布虚拟支付下单和回调处理，保留旧支付下单接口
5. 前端发布：新版本根据后端返回或配置决定支付方式，老版本继续走旧链路
6. 小流量验证：内部账号完成下单、支付、回调、会员开通完整链路
7. 分批放量：按 userId 白名单和百分比逐步扩大
8. 全量开放：监控连续稳定后再打开 100%

## 5. 监控指标
| 指标 | 观察方式 | 告警阈值 | 异常处理 |
|---|---|---|---|
| 会员支付成功率 | 支付订单统计 | 较旧链路下降超过 5% | 暂停放量 |
| 回调成功率 | 回调日志 | 错误率超过 1% | 关闭开关并排查 |
| 会员开通成功率 | 会员订单和用户会员状态对账 | 失败超过 3 单 | 人工补偿并暂停灰度 |
| 支付接口错误率 | 接口监控 | 错误率超过 1% | 回退旧链路 |
| 回调异常日志 | 日志平台 | 连续出现异常 | 暂停灰度 |

## 6. 回滚方案
- 可直接回滚内容：前端入口、后端下单选择、feature flag 配置
- 不可直接回滚内容：已创建的微信虚拟支付订单
- 回滚步骤：
  1. 关闭 enable_virtual_vip_pay
  2. 新请求全部回到旧支付链路
  3. 保留虚拟支付回调处理，继续处理已支付订单
  4. 对账确认是否存在支付成功但未开通会员的订单
- 回滚后验证：新订单走旧链路；已支付虚拟订单仍能正常开通；不存在重复续期

## 7. 数据兼容与风险点
1. 风险：支付成功但会员未开通
   - 触发条件：回调处理失败
   - 影响：用户付费后权益未到账
   - 规避方案：回调日志、订单状态、会员状态必须可对账
2. 风险：重复回调导致重复续期
   - 触发条件：微信重复通知
   - 影响：会员有效期被多次增加
   - 规避方案：以支付订单号做幂等处理
3. 风险：老版本前端不兼容
   - 触发条件：老版本调用新支付入口
   - 影响：支付失败
   - 规避方案：老版本继续走旧链路

## 8. 最小可执行方案
- 发布方式：保留旧链路，通过 feature flag 切换新链路
- 第一批灰度：内部账号 + 20 个指定用户
- 放量节奏：内部 -> 5% -> 20% -> 100%
- 回滚开关：enable_virtual_vip_pay=false
- 必看指标：支付成功率、回调成功率、会员开通成功率、回调异常日志

使用说明

复制 Prompt 后，补齐发布功能、灰度对象、开关设计、兼容策略、监控阈值和回滚方式。核心链路、外部支付、登录和状态流转类变更，建议优先按“白名单 + feature flag + 分批放量”输出方案。
