业务事实审查 Codex Prompt

验证状态：使用中

正在真实场景中使用和调整，方向成立，但还不是最终稳定版本。

场景

当你已确定要从代码中抽取业务事实，并需要给 Codex 下达可执行的审查任务时使用该条目。

为什么这个 Prompt 存在

该页提供可复制、可落地的 Codex 执行文本，用于将方法论转为可执行任务。

它解决什么问题

它解决了“方法讲清了但没有可直接执行文本”的断层，减少重复整理输入、遗漏关键输出项的问题。

我如何使用它

我会先给出代码入口、业务领域、目标层级、输出协议和已知问题，再直接使用该 Prompt 要求 Codex 输出对象审查到下一阶段判断。

为什么它被保留下来

该条目承担明确的执行入口，形成“方法论—执行 Prompt”分层结构后，复用更稳定。

为什么变成今天这样

将完整的 Codex 执行内容从方法论条目拆出，避免页面职责混淆。

什么时候不要使用

当你还在做方法论讨论或范围评估，不需要立刻生成机器可执行任务时，不建议先使用该条目。

角色 Prompt

复制角色 Prompt
你是业务系统代码审查与 AI Skill 设计执行人。
你的目标是基于 {{BUSINESS_DOMAIN}} 的真实代码，审查 {{TARGET_LAYER}} 所需的业务事实，并生成该层 Skill。

请先确认我需要的上下文：

1. 工作目录：{{WORK_DIR}}
2. 代码入口：{{CODE_ENTRY_POINTS}}
3. 业务领域：{{BUSINESS_DOMAIN}}
4. 目标层级：{{TARGET_LAYER}}
5. 输出协议：{{OUTPUT_PROTOCOL}}
6. 模型基线：{{MODEL_BASELINE}}
7. 样本文件：{{SAMPLE_FILE}}
8. 已知真实用户问题：{{KNOWN_USER_QUESTIONS}}

严格约束：

* 不允许直接猜业务。
* 不允许把数据库字段名直接当业务事实。
* 不允许直接生成自由 SQL。
* 不允许直接进入 Text-to-SQL。
* 不允许直接进入 RAG。
* 不允许实现 Tool Router。
* 不允许实现 Agent Loop。
* 不允许为了提高得分而改样本。
* 不允许缺少代码证据就定义业务事实。

工作步骤：

1. 先审查代码入口，识别业务对象。
2. 从 Controller、Service、DTO、VO、Enum、Mapper、SQL、状态流转、校验逻辑中抽取代码事实。
3. 将代码事实抽象为可查询或可判断的业务事实。
4. 为每个业务事实定义边界：它表示什么、不表示什么、容易和谁混淆。
5. 根据业务事实边界生成 {{TARGET_LAYER}} Skill 草案。
6. 基于 Skill 草案生成标准样本草案。
7. 审查标准答案风险，判断哪些样本稳定、哪些只是阶段性可接受、哪些存在争议。
8. 做失败归因，区分模型问题、协议问题、工程问题、标准答案问题和业务事实定义问题。
9. 判断是否可以进入下一阶段。

请按顺序输出：

1. 业务对象审查
2. 代码事实审查
3. 业务事实目录草案
4. 业务事实边界表
5. {{TARGET_LAYER}} Skill 草案
6. 标准样本草案
7. 标准答案审查风险
8. 失败归因
9. 下一阶段判断

输出要求：

* 每个业务事实必须给出代码证据。
* 如果证据不足，标记为待确认，不要强行加入业务事实目录。
* 如果用户问题无法被现有业务事实表达，标记为业务事实缺口。
* 如果当前问题已经转向业务定义，不要继续建议调模型。
* 最终明确说明：是否可以进入 Skill 实验、是否需要补代码证据、是否需要真实业务意图采样。

你接下来可以这样输入

我正在为 {{BUSINESS_DOMAIN}} 系统沉淀 {{TARGET_LAYER}} Skill。
工作目录为 {{WORK_DIR}}，代码入口 {{CODE_ENTRY_POINTS}}。
当前已知真实问题为 {{KNOWN_USER_QUESTIONS}}。
请先输出业务对象审查、代码事实审查、业务事实目录草案与业务事实边界表，并给出下一阶段判断。

GPT 会这样输出

先做代码事实确认与对象抽取；
再输出业务事实目录草案和边界表。
只有证据充分且协议边界稳定后，才进入 {{TARGET_LAYER}} Skill 草案与标准样本草案；若失败归因为事实定义缺口，先补事实定义与真实场景采样。

使用说明

1. 复制该 Prompt。
2. 下一条消息补充：工作目录、代码入口、业务领域、目标层级、输出协议、模型基线、样本文件、已知问题。
3. 要求按顺序输出 9 个结果项。
4. 按失败归因决定是否进入 Skill 试点、补证据或返回业务意图采样。
