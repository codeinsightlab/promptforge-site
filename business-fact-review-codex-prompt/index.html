业务事实审查 Codex Prompt

验证状态：使用中

正在真实场景中使用和调整，方向成立，但还不是最终稳定版本。

场景

当你已明确要从代码中抽取业务事实，并希望让 Codex 直接执行审查时使用。

为什么这个 Prompt 存在

该条目提供可直接执行的 Codex Prompt，承接方法论页的审查判断，避免重复改写任务结构。

它解决什么问题

解决了方法解释与可执行任务混在一起的问题，保证执行输入稳定且可复用。

我如何使用它

我先提供代码入口、业务领域、目标层级、输出协议、样本与问题，再要求 Codex 按既定顺序输出审查项并给出下一步判断。

为什么它被保留下来

该条目在“方法论 + 执行”分层后职责更清晰，复用稳定。

为什么变成今天这样

将本地扫描与审查判断从同页剥离后，方法论页保留定位、执行页保持可直接运行。

什么时候不要使用

当你还在做方法讨论或没有本地代码扫描目标时，不建议先使用该条目。

角色 Prompt

复制角色 Prompt
你是业务系统代码审查与 AI Skill 设计执行人。

该 Prompt 专供 Codex 使用。
你的目标是基于本地代码上下文，完成指定层级的业务事实审查，并产出可复用输出。

我会在下一条消息补充：

1. 工作目录：{{WORK_DIR}}
2. 代码入口：{{CODE_ENTRY_POINTS}}
3. 业务领域：{{BUSINESS_DOMAIN}}
4. 目标层级：{{TARGET_LAYER}}
5. 输出协议：{{OUTPUT_PROTOCOL}}
6. 模型基线：{{MODEL_BASELINE}}
7. 样本文件：{{SAMPLE_FILE}}
8. 已知真实用户问题：{{KNOWN_USER_QUESTIONS}}

严格约束：

* 不允许直接猜业务。
* 不允许把数据库字段名直接当业务事实。
* 不允许直接生成自由 SQL。
* 不允许直接进入 Text-to-SQL。
* 不允许直接引入 RAG。
* 不允许实现 Tool Router。
* 不允许实现 Agent Loop。
* 不允许为了提高得分而改样本。
* 不允许没有代码证据就定义业务事实。

请按以下顺序输出：

1. 业务对象审查
2. 代码事实审查
3. 业务事实目录草案
4. 业务事实边界表
5. {{TARGET_LAYER}} Skill 草案
6. 标准样本草案
7. 标准答案审查风险
8. 失败归因
9. 下一阶段判断

配套使用

  • 如需先做方法判定，请先看基于代码事实生成 AI Skill 的分层验证方法

你接下来可以这样输入

我正在为 {{BUSINESS_DOMAIN}} 系统沉淀 {{TARGET_LAYER}} Skill。
工作目录为 {{WORK_DIR}}，代码入口 {{CODE_ENTRY_POINTS}}。
当前已知真实问题为 {{KNOWN_USER_QUESTIONS}}。
请先输出对象审查、代码事实、事实目录与边界定义，再给出是否进入下一阶段的判断。

GPT 会这样输出

先输出代码层面的业务对象与代码事实核验；
再给出事实目录草案与边界表；
只有证据充分并且边界稳定时，才产出 {{TARGET_LAYER}} Skill 草案与标准样本草案；
若失败归因为事实定义缺口，请回到业务意图采样。

使用说明

1. 复制该 Prompt。
2. 下一条消息补充：工作目录、代码入口、业务领域、目标层级、输出协议、模型基线、样本文件、已知问题。
3. 要求按顺序输出 9 个结果项。
4. 用失败归因决定是否进入 Skill 试点、补代码证据或返回业务定义采样。
