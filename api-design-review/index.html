复制
## 1. 是否建议上线
- 结论：修改后上线
- 主要原因：当前接口把 draftId 暴露给前端，存在越权和重复提交风险。
- 最大风险：用户可能提交不属于自己的 draftId，或者重复点击导致多次调用汇付接口。

## 2. 接口设计问题
1. 问题：draftId 不应作为小程序端提交入口的唯一依据
   - 证据：请求参数直接传 draftId
   - 影响：如果后端校验不足，可能出现越权提交
   - 修改建议：提交接口不接收 draftId，后端根据当前登录 userId 查询唯一草稿
2. 问题：缺少重复提交保护
   - 证据：暂无 requestId，也未说明状态锁定
   - 影响：重复点击或网络重试可能多次调用汇付
   - 修改建议：基于 userId + 当前草稿状态做提交保护，提交中或已提交状态直接拒绝或返回当前结果

## 3. 幂等性评估
- 是否具备幂等性：不完整
- 当前幂等依据：user_id 唯一草稿
- 缺失点：没有提交中状态、没有外部请求流水号、没有重复调用处理
- 建议幂等方案：以 userId 查询草稿并加状态校验；每次正式提交生成 submitLogId；外部请求使用唯一 requestNo

## 4. 安全性评估
- 登录态是否可信：应以后端登录态为准
- userId 是否允许前端传：不允许
- 权限是否足够：需要校验草稿 userId == 当前登录 userId
- 金额 / 状态等敏感字段是否由后端控制：应由后端控制
- 是否存在越权风险：存在，需要调整

## 5. 数据一致性与事务评估
- 是否需要事务：需要
- 事务边界建议：草稿完整性校验、提交日志创建、状态更新应在事务内完成
- 是否涉及外部调用：涉及汇付
- 外部调用失败如何处理：记录失败日志并返回明确错误
- 是否需要补偿或重试：建议保留提交日志，为后续人工排查或重试提供依据

## 6. 扩展性评估
- 当前设计是否容易扩展：一般
- 哪些字段可能需要预留：requestNo、submitStatus、externalResponseCode、externalResponseMsg
- 哪些设计不建议过早抽象：不建议现在抽象通用商户入驻流程，先保证个人入驻主链稳定

## 7. 最小修改方案
- 请求参数需要调整：submit 接口不再接收 draftId
- 后端校验需要增加：根据当前 userId 查询草稿；校验资料完整性；校验状态是否允许提交
- 数据库约束需要增加：保留 user_id 唯一；提交日志增加 request_no
- 事务处理需要调整：日志创建和本地状态更新在事务内；外部调用异常必须可追踪
- 响应结构需要调整：返回 submitLogId、submitStatus、明确错误信息
- 风险等级：中

## 8. 建议后的接口形态
请求：
POST /api/hfpay/personal-merchant-entry/submit
{}

后端关键逻辑：
1. 获取当前登录 userId
2. 根据 userId 查询唯一草稿
3. 校验草稿存在且归属当前用户
4. 校验草稿资料完整
5. 创建提交日志 requestNo
6. 调用汇付入驻接口
7. 根据结果更新日志和草稿状态
8. 返回提交结果
