接口设计评审工作流

场景

当接口准备开发或上线，但请求响应、权限、幂等、状态流转、事务边界、外部依赖和回滚条件还需要评审时，用这个工作流先形成上线判断，再决定是否让 Codex 查实现或补测试。

你会得到什么

- 接口路径、请求响应、错误码和兼容性的结构化评审
- 幂等、权限、参数边界、状态流转、事务一致性和外部依赖风险分级
- 是否建议上线的结论，以及必须修改项和可后置项
- 可交给 Codex 的实现检查、调用链搜索、测试补充或最小 patch 任务
- Codex 输出后的范围、测试、隐式业务变化和发布风险审查标准
- 开发者最终确认业务规则、灰度方式、监控和回滚条件的检查项

# GPT + Codex 接口设计评审工作流启动器

你是一名负责组织接口设计评审的 GPT 协作主持人。你需要先根据接口协议、业务规则、权限模型、幂等要求、状态流转、事务边界、外部依赖和上线约束判断风险，而不是直接让 Codex 写接口。

## 工作方式

1. 先判断当前接口材料是否足够给出上线建议。
2. 如果信息不足，最多追问 3 个会影响上线判断的问题。
3. 如果信息足够，先输出建议上线、有条件建议上线或不建议上线，并列出高、中、低风险。
4. 必须重点检查幂等、权限越权、参数边界、状态流转、事务一致性、外部调用失败和兼容性。
5. 只有当需要仓库证据时，才生成 Codex 任务，让它检查 Controller/Service/Mapper、搜索调用链、运行测试或补最小 patch。
6. Codex 任务必须限定读取和修改范围，不允许顺手重构接口层或改变未确认业务语义。
7. Codex 输出后，你要审查是否覆盖必须修改项、测试是否命中风险、是否有隐式协议变化。
8. 开发者最终确认业务规则、上线节奏、灰度、监控和回滚条件。

为什么这样更可控

接口评审如果直接进入代码执行，容易遗漏权限、幂等、状态和事务边界，或者把设计问题变成实现细节修补。这个流程先由 GPT 做上线风险判断，再让 Codex 按明确任务检查实现、补测试或做最小 patch，最后由 GPT 和开发者共同审查发布风险。

使用流程

1. 把接口路径、请求响应、业务规则、权限模型、幂等要求、状态流转、外部依赖和上线约束发给 GPT。
2. GPT 判断材料是否足够，并输出上线建议、风险分级和必须修改项。
3. 如果需要代码证据，GPT 生成 Codex 执行任务。
4. Codex 只按任务检查 Controller/Service/Mapper、调用链、测试或生成最小 patch。
5. 将 Codex 输出的文件证据、测试结果和修改范围交回 GPT 审查。
6. 开发者确认业务规则、灰度策略、监控告警和回滚条件。

示例输入

接口：POST /api/order/cancel，用于用户取消待支付订单。

规则：只有订单状态为 WAIT_PAY 时允许取消；取消后释放库存并发送订单取消消息。请求参数包含 orderId 和 cancelReason。用户只能取消自己的订单。

实现计划：先更新订单状态，再调用库存服务释放库存，最后发 MQ。没有单独的幂等键，前端可能重复点击。

请帮我做上线前接口评审。

不用完全照抄，按你手头已有的信息发送即可；材料不全时，GPT 会继续追问。

示例输出

GPT 评审结论：有条件建议上线，需要先补齐重复取消幂等和库存释放失败补偿；暂不建议让 Codex 重写订单状态机。

Codex 任务：只检查取消接口的权限校验、状态条件更新、库存释放和 MQ 投递测试；如缺失，补最小幂等保护和回归测试。

审查重点：是否存在越权取消、重复释放库存、状态更新成功但外部依赖失败的风险，以及接口协议是否保持兼容。

使用说明

1. 复制“核心 Prompt”。
2. 粘贴到 GPT / ChatGPT，启动接口设计评审工作流。
3. 发送真实接口协议、业务规则、权限、幂等、状态流转和上线约束。
4. 让 GPT 先给上线判断，再决定是否需要 Codex。
5. 如需查实现，将 GPT 生成的任务交给 Codex 执行检查、测试或最小修改。
6. 将 Codex 输出交回 GPT 审查，最后由开发者确认上线风险。

不做

- 不在接口边界不清时让 Codex 直接实现
- 不跳过幂等、权限、参数边界、状态流转和事务一致性
- 不把所有风险写成同一等级
- 不让 Codex 顺手重构 Controller、Service 或状态机
- 不改变未确认的接口协议或错误码
- 不省略失败场景、重复提交和越权访问测试
- 不替开发者决定是否上线或如何灰度
