# 分析约束 / 执行约束

1. 必须先判断是否真的需要分布式拆分，不要为了分布式而分布式。
2. 必须明确服务边界、数据归属、跨服务调用和一致性策略。
3. 必须区分同步链路、异步链路、核心链路和非核心链路。
4. 必须覆盖故障隔离、容量、观测、灰度、回滚和迁移成本。
5. 不要只给概念性架构图描述，必须给出可执行的阶段方案和风险控制。
6. 信息不足时最多追问 3 个关键问题。

结论：不建议一次性拆成订单、库存、商品、支付四个独立服务；应先稳定商品只读能力和库存锁定边界，再拆订单主流程。

关键风险：订单创建链路会引入跨服务一致性和超时未知问题，库存锁定必须有幂等键、过期释放和补偿任务。
