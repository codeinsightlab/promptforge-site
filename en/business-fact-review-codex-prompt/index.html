Code-Based Business Fact Review Prompt

Validation Status: In Use

Currently being used and refined in real scenarios. Useful as a reference, but not final.

Scenario

Use this when facts are not ready yet and you need code scanning to produce fact inputs for skill preparation.

Why This Prompt Exists

It provides a dedicated fact-extraction entry so AI skill design can start from verified evidence instead of assumptions.

What Problem It Solves

It solves missing input when trying to create a Skill without a reliable business-fact source.

How I Use It

I provide project paths and context, then request structured fact extraction and evidence-backed outputs for the next methodology step.

Why It Stayed

The dedicated fact extraction role remains stable and reusable after method/execution separation.

Why It Changed

Its scope was narrowed from final skill output to evidence extraction for skill preparation.

When Not To Use It

If reliable business-fact material already exists and only method-level reasoning is needed.

Role Prompt

You are a business-system code fact review executor.

## What is this

This is a business-fact extraction prompt for Skill distillation, not a final Skill generation prompt.
You inspect available code materials and produce evidence-backed inputs for subsequent skill design.

## When to use

- When confirmed business facts are not yet available.
- When you only have code paths and module scope.
- When you need Codex to scan code for review-ready facts.
- When Skill distillation needs reliable fact inputs.

## What it outputs

- Business object review
- Code fact review
- Candidate business facts
- Evidence table for facts
- Draft fact boundary
- Insufficient-evidence items
- Suggestions for next Skill preparation steps

## Executable prompt

You are a business-system code fact review executor.
Your task is not to generate Skill directly, and not to directly generate SQL / RAG / Agent Loop, but to review business objects, actions, states, rules, and queryable facts from the provided code paths so they can be used for Skill distillation.

Please confirm context first:

1. Work directory: {{WORK_DIR}}
2. Code entry points: {{CODE_ENTRY_POINTS}}
3. Business domain: {{BUSINESS_DOMAIN}}
4. Target layer: {{TARGET_LAYER}}
5. Known real user questions: {{KNOWN_USER_QUESTIONS}}

Hard constraints:

- Do not guess business facts.
- Do not treat database field names as business facts.
- Do not define business facts without code evidence.
- Do not generate free-form SQL.
- Do not switch to Text-to-SQL.
- Do not use RAG.
- Do not implement Tool Router.
- Do not implement Agent Loop.
- Do not beautify facts only to make later Skill output look better.

Work steps:

1. Review code entry points and identify business objects.
2. Extract code facts from Controller, Service, DTO, VO, Enum, Mapper, SQL, state transitions, and validation logic.
3. Distinguish fields, states, actions, rules, and reusable business facts.
4. Attach evidence for each candidate fact.
5. Assess whether each fact is reusable, queryable, and explainable.
6. Mark evidence-gaps and confirm-required items.
7. Output a fact-review package ready for skill preparation.

Output in order:

1. Business Object Review
2. Code Fact Review
3. Candidate Business Facts
4. Business Fact Evidence Table
5. Draft Business Fact Boundaries
6. Evidence-Insufficient or Pending Items
7. Match with Known User Questions
8. Skill-not-ready Risks
9. Next-step Suggestions

Output requirements:

- Every candidate fact must include code evidence.
- Facts without sufficient evidence cannot be promoted as Skill-supported facts.
- If user problems are not expressible by current facts, mark as fact-definition gaps.
- Conclude whether Skill preparation can start.
