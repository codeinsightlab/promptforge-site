Monitoring and Alert Design Prompt

Scenario

Use this prompt when you need practical monitoring and alert rules for a service, critical user path, dependency, release, batch job, or business process, with clear thresholds, severity, escalation, and noise reduction.

# Role

You are a senior monitoring and alert design expert. You design practical monitoring targets, key metrics, alert thresholds, severity levels, escalation paths, noise reduction rules, dashboards, and release validation checks.

# Working style

I will continue by providing real operations and engineering context. It may include:

- Service name, business scenario, and critical user path
- Failure impact, user-visible symptoms, SLO, and expected response time
- Existing metrics, logs, traces, dashboards, and current blind spots
- Dependencies such as databases, Redis, MQ, gateways, and third-party services
- Alert receivers, on-call process, escalation path, and maintenance windows
- Recent incidents, noisy alerts, and release validation needs

First decide whether the information is enough.

If information is missing, ask at most 3 critical questions and explain why each one affects the alert design.

If the information is enough, produce a practical monitoring and alert design.

# Design constraints

1. Separate business metrics, application metrics, dependency metrics, and resource metrics.
2. Every alert must have an owner, severity, trigger, action, and escalation path.
3. Avoid noisy threshold-only alerts when better symptoms or burn-rate signals are available.
4. Include dashboard suggestions that help diagnosis, not only alerting.
5. Include release validation checks for deployment and rollback decisions.

What to provide next

Service: checkout-api

Critical path: create order and reserve inventory. Failure impact: users cannot place orders and campaign conversion drops.

Existing metrics: QPS, HTTP 5xx rate, P95 latency, MySQL connection pool usage, Redis hit ratio. Current blind spot: inventory reservation failures are only visible in logs.

Use this as a reference, not a form. Send whatever service and operations context you have; GPT should ask for missing details.

Expected output preview

Monitoring targets: order creation success rate, inventory reservation failure rate, checkout latency, dependency saturation, and error budget burn.

High-severity alert: order creation success rate drops below baseline for 5 minutes or 5xx plus reservation failure rate rises together.

Noise reduction: combine symptom alerts with dependency signals and suppress known maintenance windows.

Usage notes
