Scenario

What You Get

- A review conclusion for Redis keys, TTL, value structure, data volume, and hot-key risk

- Required changes before release, deferrable improvements, and monitoring metrics

# GPT + Codex Redis Usage Review Workflow Starter

You are the GPT coordinator for a controlled Redis usage review. First reason from the business scenario, key design, TTL, data structure, read/write frequency, value size, consistency requirement, database fallback, and monitoring signals. Do not ask Codex to change cache code immediately.

## Working method

1. Decide whether the current Redis design material is enough to assess release risk.
2. If information is missing, ask at most 3 critical questions, such as key cardinality, value size, TTL strategy, hot-key QPS, update behavior, or database fallback.
3. If information is enough, output overall risk level, main risks, trigger conditions, impact scope, and required changes.
4. Always check penetration, breakdown, avalanche, hot keys, big keys, blocking commands, memory growth, consistency, and degradation path.
5. Generate a Codex task only when repository evidence is needed: cache call-chain inspection, configuration checks, key construction, TTL settings, tests, or a minimal patch.
6. The Codex task must constrain read and change scope. Do not turn one Redis review into a cache-framework refactor.
7. After Codex reports back, review whether the implementation meets the risk-control target, whether tests cover high-risk paths, and whether monitoring is observable.
8. The developer confirms business consistency, capacity budget, degradation strategy, rollout timing, and rollback conditions.

Why This Is More Controlled

Redis review becomes risky when AI jumps straight into code and turns key, TTL, or consistency problems into local patches or a cache-framework refactor. This workflow makes GPT assess value, risk, and evidence gaps first, then lets Codex perform controlled call-chain inspection, configuration checks, tests, or a minimal patch before capacity, fallback, and release risks are confirmed.

Usage Flow

1. Send business scenario, key design, value structure, TTL, QPS, data volume, update behavior, consistency needs, and database fallback to GPT.
2. GPT decides whether the material is enough and outputs risk level, required changes, and monitoring verification.
3. If code evidence is needed, GPT writes a Codex execution task.
4. Codex only inspects cache call chains, key construction, TTL configuration, tests, or creates a minimal patch within that task.
5. Return Codex file evidence, test results, configuration changes, and risks to GPT for review.
6. The developer confirms business consistency, capacity budget, degradation strategy, rollout timing, and rollback plan.

Example Input

Key pattern: product:detail:{productId}

Value: JSON product detail, usually 3 KB but can reach 80 KB for complex products. TTL is 30 minutes. The cache is deleted after product updates.

Traffic: the top 100 products receive most reads during campaigns. Missing cache falls back to MySQL.

Use this as a reference, not a form. Send whatever Redis material you have; GPT should ask for missing details.

Example Output

GPT review conclusion: product base information can be cached, but price should not be tightly coupled to a 30-minute product-detail JSON; hot-key and breakdown risk are medium-high during campaigns.

Codex task: inspect only product-detail cache keys, TTL settings, invalidation flow, and related tests; if missing, add minimal TTL jitter or hot-key protection.

Review focus: avoiding cache-framework refactors, keeping price consistency explicit, and verifying database fallback plus monitoring metrics.

Usage

1. Copy the Core Prompt.
2. Paste it into GPT or ChatGPT to start the Redis usage review workflow.
3. Send real key design, TTL, data structure, QPS, update behavior, consistency, and monitoring context.
4. Let GPT decide the risk and whether Codex is needed.
5. If implementation evidence is needed, send the GPT-generated task to Codex for call-chain checks, configuration checks, tests, or a minimal patch.
6. Return Codex output to GPT for review, then confirm release risk yourself.

Do Not

- Do not assume all data should be cached
- Do not skip key cardinality, value size, TTL, hot-key, and consistency evidence
- Do not let Codex turn one review into a cache-framework refactor
- Do not add complex abstractions for hypothetical future scenarios
- Do not leave database fallback, degradation, and monitoring as generic advice
- Do not claim the design is safe without capacity and hit-ratio evidence
- Do not delegate business consistency tradeoffs or rollout timing to AI
