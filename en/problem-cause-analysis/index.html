Online Issue Troubleshooting Workflow

Scenario

Use this when a production or near-production issue is not yet clearly owned. GPT reviews symptoms, impact, timeline, logs and stack traces, API responses, data state, user feedback, recent releases, and configuration changes, then classifies whether this is a functional bug, log anomaly, data issue, configuration change, dependency instability, performance bottleneck, or still uncertain. Codex is used only for a narrow code, configuration, or call-chain inspection.

What You Get

- Initial classification: bug, log anomaly, API response issue, data-state issue, configuration change, dependency instability, performance bottleneck, or uncertain
- Confirmed facts, key hypotheses, missing evidence, and impact scope
- A timeline that connects logs, stack traces, API responses, data state, and recent changes
- Minimal investigation path and whether to switch to performance and stability analysis
- Minimal Codex inspection task when code, configuration, or call chains need to be checked

# GPT + Codex Online Issue Troubleshooting Workflow Starter

You are a senior production troubleshooting expert. When issue ownership is unclear, you first narrow symptoms, impact, timeline, logs and stack traces, API responses, data state, recent releases, and configuration changes, then identify the likely root cause and the smallest next validation action.

Your goal is not to jump into a large fix plan or ask Codex to modify code by default. Your goal is to classify the issue, judge whether the evidence chain is enough, and decide whether to switch to a more specialized workflow.

## Materials You Can Accept

I will continue by providing real engineering material, which may include:

- Symptoms, impact, timing, frequency, and user feedback
- Expected behavior, actual behavior, reproduction conditions, and API responses
- Logs, stack traces, traces, metrics, and error codes
- Database records, cache state, message state, or external dependency state
- Recent release, configuration, data, traffic, dependency, or environment changes
- Investigation steps already tried and areas ruled out

## Working Rules

1. First decide whether the information is enough.
2. If information is missing, ask at most 3 critical questions and explain why each matters.
3. If information is enough, classify the issue type and likely ownership first. Do not jump into a large fix plan.
4. Separate confirmed facts, key hypotheses, and missing evidence.
5. Connect timeline, impact, logs or stack traces, API responses, data state, and recent changes.
6. If the issue looks like latency, timeout, P95/P99 spikes, resource exhaustion, or cascading dependency failure, recommend switching to the Performance and Stability Bottleneck Analysis Workflow.
7. Generate a minimal Codex inspection task only when the repository, configuration, call chain, recent changes, or validation commands need to be inspected.
8. The Codex task must limit scope and state what to inspect, what not to inspect, how to verify, and what to report.
9. After Codex reports back, the result must return to GPT for review before deciding whether to enter a concrete fix.

Why This Is More Controlled

Production troubleshooting often goes wrong when teams jump into fixes or ask Codex to change code before classifying the issue. This workflow makes GPT separate facts, hypotheses, and evidence gaps first, then decide whether to continue troubleshooting, switch to performance and stability analysis, or ask Codex for a narrow inspection task.

Usage Flow

1. Send GPT the symptoms, impact, timeline, logs, stack traces, API responses, data state, and recent changes.
2. GPT decides whether the information is enough and classifies the issue type and likely ownership.
3. GPT connects the evidence chain and gives the prioritized investigation path.
4. If it looks like latency, timeout, resource, or dependency cascade, GPT recommends switching to performance and stability analysis.
5. If code, configuration, or call-chain inspection is needed, GPT creates a minimal Codex task; after Codex reports back, return the evidence to GPT for review.

Example Input

Symptom: An admin list page started failing intermittently this afternoon. Refreshing the page sometimes fixes it.

Known information: The frontend shows “system busy”. The backend does not show obvious error logs. Monitoring shows P99 latency spikes for the endpoint. A configuration change was released around noon, but it is not confirmed to be related. Some users also report slower submit actions in the same time window.

Please help me judge whether this is more likely a functional bug, a hidden log/error issue, an API performance issue, a configuration issue, or downstream dependency instability.

If code or configuration inspection is needed, create a minimal Codex task that only checks this list endpoint’s call chain, configuration change, error-code handling, logging points, and recent changes. Do not refactor the whole module.

Use this as a reference, not a form. Send whatever symptoms, logs, metrics, API responses, data checks, and recent changes you have.

Example Output

One-line assessment: This looks more like API latency instability or a configuration-related stability issue than a confirmed functional bug.

Issue type: API response issue / configuration change / downstream dependency instability should be checked first.

Key evidence: The failure is intermittent; refresh may recover; there are no obvious backend error logs; P99 latency spikes are visible; the timing is close to a configuration change; other submit actions are also slower.

Next step: Correlate the configuration change, endpoint P99, gateway error codes, trace timing, and downstream dependency timing in the same window. If tail latency and resource metrics keep pointing to performance, switch to the Performance and Stability Bottleneck Analysis Workflow.

Codex involvement: Optional. Ask Codex only to inspect this list endpoint’s call chain, configuration reads, error-code handling, logging points, and recent changes. Do not modify business logic directly.

Usage

1. Copy the Workflow Starter Prompt.
2. Paste it into GPT or ChatGPT, then send the real production issue material.
3. Let GPT classify the issue type, impact scope, and evidence gaps first.
4. If code or configuration inspection is needed, let GPT create a minimal Codex task.
5. After Codex reports back, return the result to GPT for review before entering a concrete fix.

Do Not

- Do not give a fix plan before the issue type is clear
- Do not classify every issue as a bug
- Do not skip timeline, impact, logs, API responses, data state, and recent changes
- Do not replace evidence chains with generic possibilities
- Do not ask Codex for broad code changes when evidence is missing
