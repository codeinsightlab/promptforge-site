API Design Review Workflow

Scenario

Use this workflow before API implementation or release when request and response shape, permissions, idempotency, state transitions, transaction boundaries, external dependencies, and rollback conditions still need review.

What You Get

- A structured review of endpoint path, request and response shape, error codes, and compatibility
- Risk grading for idempotency, permission boundaries, parameter validation, state transitions, transaction consistency, and external dependencies
- A release recommendation with must-fix items and deferrable improvements
- A controlled Codex task for implementation inspection, call-chain search, test additions, or a minimal patch
- Review criteria for Codex output covering scope, tests, implicit business changes, and release risk
- Developer decision points for business rules, rollout method, monitoring, and rollback conditions

# GPT + Codex API Design Review Workflow Starter

You are the GPT coordinator for a controlled API design review. First reason from the API contract, business rules, permission model, idempotency needs, state transitions, transaction boundaries, external dependencies, and release constraints. Do not ask Codex to implement the API immediately.

## Working method

1. Decide whether the current API material is enough to give a release recommendation.
2. If information is missing, ask at most 3 questions that affect release judgment.
3. If information is enough, state whether the API is ready, conditionally ready, or not ready, and classify high, medium, and low risks.
4. Always review idempotency, privilege boundaries, parameter validation, state transitions, transaction consistency, external-call failures, and compatibility.
5. Generate a Codex task only when repository evidence is needed: Controller/Service/Mapper inspection, call-chain search, tests, or a minimal patch.
6. The Codex task must constrain read and change scope. Do not let Codex refactor the API layer casually or change unconfirmed business semantics.
7. After Codex reports back, review whether must-fix items are covered, tests hit the risks, and any implicit contract changes appeared.
8. The developer confirms business rules, rollout timing, gray release, monitoring, and rollback conditions.

Why This Is More Controlled

API review becomes risky when code execution starts before permissions, idempotency, state, and transaction boundaries are clear. This workflow makes GPT produce the release-risk judgment first, then lets Codex inspect implementation, add tests, or create a minimal patch within a defined task, with GPT and the developer reviewing release risk afterward.

Usage Flow

1. Send endpoint path, request and response shape, business rules, permission model, idempotency needs, state transitions, external dependencies, and release constraints to GPT.
2. GPT decides whether the material is enough and outputs a release recommendation, risk grading, and must-fix items.
3. If code evidence is needed, GPT writes a Codex execution task.
4. Codex only inspects Controller/Service/Mapper code, call chains, tests, or creates a minimal patch within that task.
5. Return Codex file evidence, test results, and change scope to GPT for review.
6. The developer confirms business rules, rollout strategy, monitoring alerts, and rollback conditions.

Example Input

Endpoint: POST /api/orders/{orderId}/cancel

Caller: mobile app. QPS is low, but users may tap repeatedly. The request body contains cancelReason and refundMethod. Only the order owner can cancel.

The API updates order status, creates a refund request, and calls the payment provider asynchronously.

Use this as a reference, not a form. Send whatever API material you have; GPT should ask for missing details.

Example Output

GPT review conclusion: conditionally ready, but repeated-cancel idempotency and inventory-release failure compensation must be addressed first; Codex should not rewrite the order state machine.

Codex task: inspect only the cancel endpoint permission check, conditional status update, inventory release, MQ dispatch, and tests; add minimal idempotency protection and regression tests if missing.

Review focus: horizontal privilege escalation, duplicate inventory release, external dependency failure after status update, and whether the API contract remains compatible.

Usage

1. Copy the Core Prompt.
2. Paste it into GPT or ChatGPT to start the API design review workflow.
3. Send the real API contract, business rules, permissions, idempotency, state transitions, and release constraints.
4. Let GPT give the release judgment before deciding whether Codex is needed.
5. If implementation evidence is needed, send the GPT-generated task to Codex for inspection, tests, or a minimal patch.
6. Return Codex output to GPT for review, then confirm release risk yourself.

Do Not

- Do not ask Codex to implement while API boundaries are unclear
- Do not skip idempotency, permissions, parameter boundaries, state transitions, and transaction consistency
- Do not flatten all risks into the same severity
- Do not let Codex casually refactor Controller, Service, or state-machine code
- Do not change unconfirmed API contracts or error codes
- Do not omit failure, repeated-submit, and privilege tests
- Do not delegate release or gray-rollout decisions to AI
