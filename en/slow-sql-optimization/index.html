Slow SQL Optimization Workflow

Scenario

Use this workflow when you have a slow query, EXPLAIN output, slow logs, schema, or latency data, but still need to determine whether the bottleneck is indexing, sorting, lookups, pagination, or application query behavior.

What You Get

- Evidence-based diagnosis of SQL slowness, execution plan, and data distribution
- Analysis of index order, sorting, pagination, back-to-table lookups, temporary tables, and scanned rows
- A minimal optimization path across index changes, SQL rewrites, pagination changes, or caller parameters
- A controlled Codex task for mapper/DAO inspection, SQL rewrite, or tests
- Review criteria for Codex output using EXPLAIN, latency comparison, and rollback risk
- Developer decision points for write cost, release window, and business query semantics

# GPT + Codex Slow SQL Optimization Workflow Starter

You are the GPT coordinator for a controlled slow SQL optimization workflow. First reason from the SQL, EXPLAIN, schema, indexes, data volume, slow logs, and business query semantics before deciding whether Codex should inspect code or produce a minimal change.

## Working method

1. Decide whether the current material is enough to analyze the execution plan and business query semantics.
2. If information is missing, ask at most 3 critical questions, such as full SQL, EXPLAIN, schema, indexes, data volume, pagination style, or production latency.
3. If information is enough, identify the likely cause: index miss, excessive scanned rows, filesort, temporary table, lookup cost, deep pagination, function conversion, or caller behavior.
4. Provide the minimal optimization direction and verification method. Do not default to only saying add an index.
5. Generate a Codex task only when mapper/DAO inspection, caller checks, dynamic SQL review, tests, or a minimal patch are needed.
6. The Codex task must require code evidence, SQL changes, verification commands, and rollback concerns.
7. After Codex reports back, review EXPLAIN, latency, index cost, and business semantics evidence.
8. The developer confirms index rollout timing, write cost, rollback plan, and query-result semantics.

Why This Is More Controlled

Slow SQL work becomes risky when AI blindly adds indexes, changes query semantics, or ignores write cost. This workflow makes GPT reason through the execution plan and business constraints first, then lets Codex perform controlled call-site inspection, SQL rewrite, tests, or a minimal patch, with EXPLAIN and latency evidence reviewed afterward.

Usage Flow

1. Send SQL, EXPLAIN, schema, indexes, data volume, slow logs, QPS, and business constraints to GPT.
2. GPT decides whether the material is enough and outputs the slowness cause, minimal optimization path, and verification method.
3. If repository evidence is needed, GPT writes a Codex execution task.
4. Codex only inspects mapper/DAO code, dynamic SQL, caller parameters, tests, or creates a minimal patch within that task.
5. Return Codex SQL changes, file evidence, command results, and risks to GPT for review.
6. The developer confirms query semantics, index rollout window, write cost, and rollback plan.

Example Input

SQL: SELECT * FROM orders WHERE user_id = 18277 AND status IN (1,2) ORDER BY created_at DESC LIMIT 20;

The table has 80 million rows. Current indexes are PRIMARY(id), idx_user_id(user_id), and idx_status(status). EXPLAIN shows type=ref on idx_user_id, rows=180000, Extra=Using where; Using filesort.

The query is used on the order list page and must stay under 200 ms.

Use this as a reference, not a form. Send whatever database material you have; GPT should ask for missing details.

Example Output

GPT initial judgment: the query uses only the user_id single-column index, then scans many rows and performs filesort; do not start by refactoring pagination infrastructure.

Codex task: inspect only the order-list mapper, dynamic SQL conditions, and related tests; if needed, produce a minimal SQL or index recommendation without changing unrelated queries.

Review focus: whether EXPLAIN rows and Extra improve, whether P95 latency drops, and whether composite-index write cost and rollback are clear.

Usage

1. Copy the Core Prompt.
2. Paste it into GPT or ChatGPT to start the slow SQL workflow.
3. Send real SQL, EXPLAIN, schema, indexes, data volume, and business constraints.
4. Let GPT decide the optimization direction and whether Codex is needed.
5. If repository inspection is needed, send the GPT-generated task to Codex for SQL call-site checks or a minimal patch.
6. Return Codex output to GPT for review, then confirm rollout and rollback risk yourself.

Do Not

- Do not only say add an index without column order and evidence
- Do not skip EXPLAIN, schema, indexes, data volume, and business semantics
- Do not let Codex refactor DAO, pagination infrastructure, or multiple query modules casually
- Do not claim success without performance evidence
- Do not change query-result semantics for speed
- Do not ignore index write cost, storage cost, and rollback plan
- Do not delegate production index rollout timing to AI
