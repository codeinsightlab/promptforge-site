Code-Fact-to-Skill Methodology

Validation Status: In Use

Currently being used and refined in real scenarios. Useful as a reference, but not final.

Scenario

Useful when an existing business system already has code context and the team needs a disciplined fact-first path before any model optimization.

Why This Prompt Exists

Most projects start with prompts and models, then discover instability because facts were never defined. This method restores a fact-first sequence and keeps decisions auditable.

What Problem It Solves

It separates model, protocol, engineering, gold-set, and fact-definition failures, so teams stop confusing score tuning with missing business facts.

How I Use It

I confirm entry points, target layer, key objects, known questions, and output protocol, then move through business object review, fact extraction, fact catalog and boundary checks, and then draft Skill and gold set.

Why It Stayed

In repeated use cases, it keeps discussions anchored to evidence and reduces speculative model-level tuning before foundation facts are stable.

Why It Changed

The flow is now stage-gated around fact and protocol boundaries, replacing direct prompt or score optimization as the default first step.

When Not To Use It

Do not use it for one-off creative drafting when there is no shared code-grounded context and no need for fact-governed Skill design.

## What this is

This is a methodology for building AI Skills from code facts. It is for Tech Leads and engineering decision makers to judge whether a layer skill can be grounded on real business facts before model optimization.

## When to use

1. The business system already has code context and AI capability plans.
2. You need a semantic layer before Text-to-SQL.
3. You need to extract business facts from code and define fact boundaries.
4. You need outputs for Fact Catalog, Fact Boundary, Skill draft, and Gold Set draft.
5. You want to separate model, engineering, gold-set, and fact-definition issues.

## Core principles

1. Do not ask first how to make the model more precise; ask what real business facts are proven by code.
2. A field name is not the same as a business fact.
3. A Skill is a semantic wrapper of code facts, not a generic Prompt trick.
4. High hit rate of samples does not automatically mean business correctness.

## Method flow

1. Code review
2. Business object identification
3. Extraction of business actions, states, and rules
4. Abstraction of queryable or judgeable facts
5. Draft of fact catalog
6. Fact boundary table
7. Skill draft
8. Draft Gold Set
9. Model and execution-path evaluation
10. Gold review
11. Failure Attribution
12. Close or move to next stage

## Failure attribution

- JSON pass rate is low: check protocol control and response structure first.
- Protocol validity is low: check schema consistency, field names, and types.
- Control drift is high: no-think / JSON-only behavior is unstable.
- Semantic direction is off: fact boundaries or skill goals are drifting.
- Gold set hits but business is not right: gold answers may have issues.
- Too much ambiguity: this may come from fact-definition gaps, not model reasoning.
- SQL or field leakage: layer boundary was not cleanly separated.
