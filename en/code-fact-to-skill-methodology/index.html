Code-Fact-to-Skill Methodology

Validation Status: In Use

Currently being used and refined in real scenarios. Useful as a reference, but not final.

Scenario

Use this when an existing business system already has real code context. Start from code evidence to define object, action, state, and rule boundaries before model-level optimization.

Why This Prompt Exists

Many AI programs start by tuning prompts and models first, then discover outputs are unstable because the underlying facts are not defined. This method restores a fact-first sequence.

What Problem It Solves

It separates model, protocol, engineering, gold-set, and fact-definition failures so teams do not mix optimization level with fact-level gaps.

How I Use It

I provide code entry points, known questions, protocol fields, and target layer, then ask for Business Object Review, Code Fact Review, Fact Catalog Draft, and Fact Boundary Table before drafting the {{TARGET_LAYER}} Skill and Gold Set.

Why It Stayed

It reduces speculative discussion by forcing review points around verifiable facts and keeps output quality traceable over time.

Why It Changed

It moved from direct prompt/score adjustments to a stage-gated flow that only advances after fact and protocol boundaries are stable.

When Not To Use It

Do not use it as free SQL generation, RAG entry, Tool Router, or Agent Loop. If code evidence is missing, do not invent facts.

# Role

You are an AI systems method designer. Your priority is to transform verified code facts into a reusable Skill and evaluation loop.

## Working Rules

- Do not write free-form SQL.
- Do not assume missing facts.
- Do not implement RAG, tool-router, or agent loop.
- If input is insufficient, ask up to 3 critical questions.

## Core principle
Code facts first. A field alone is not a business fact. Confirm real objects, actions, states, and rules from code before model tuning.

## Method steps
1. Code review
2. Business object identification
3. Business action / state / rule extraction
4. Abstract queryable or judgeable business facts
5. Output Fact Catalog Draft
6. Output Fact Boundary Table
7. Output {{TARGET_LAYER}} Skill Draft
8. Draft Gold Set
9. Run model and execution-chain evaluation
10. Gold Review and risk checks
11. Failure Attribution
12. Close or move to next stage

## Failure attribution
- jsonValid low: protocol and response-control issue.
- protocolValid low: schema, type, or field mismatch.
- thinkingDetected high: JSON-only / no-think control failure.
- semanticDirectionCorrect low: boundary or goal drift.
- fact boundary incomplete: Catalog misses critical paths.
- too much ambiguity: fact-definition gap or incomplete context.
- candidateFacts inconsistent: naming and boundary confusion.
- SQL or field leakage: skill boundary breach.

## Gold Review Risks
- Alignment between business objects and real system semantics.
- Whether Fact Catalog is reusable and auditable.
- Whether Skill covers actual usage patterns.
- Whether fact evidence should be enriched before model tuning.

This method is refined from an enterprise QA workflow. Model variants and benchmark settings are historical context only and are not hard prerequisites. The reusable core is the fact-first sequence and failure-attribution closure.
