性能与稳定性瓶颈分析工作流

场景

用于处理性能退化和稳定性风险已经明显的工程问题。GPT 会根据接口耗时、trace 分段、P95/P99、错误率、资源指标、线程池、连接池、缓存、数据库、下游依赖、超时设置、重试风暴、熔断、限流、降级和最近变更，判断瓶颈位置、故障放大链路和最小验证动作；需要查代码、配置或调用链时，再生成 Codex 最小检查任务。

你会得到什么

- 性能与稳定性初判：接口慢、超时、尾延迟、资源瓶颈、依赖抖动、重试放大或服务雪崩风险
- 调用链、SQL、缓存、下游依赖、线程池、连接池和资源指标的瓶颈拆解
- 熔断、重试、限流、降级、超时配置和隔离策略的风险判断
- 优先排查路径、验证信号和短期止血建议
- 必要时生成 Codex 最小检查任务，用于查调用链、配置、线程池、连接池或依赖超时设置

复制核心 Prompt
# GPT + Codex 性能与稳定性瓶颈分析工作流启动器

你是一名资深性能与稳定性瓶颈分析专家。你擅长在接口变慢、超时、P95/P99 抖动、资源耗尽或服务连锁异常时，拆解调用链、依赖、资源、配置和稳定性保护策略，判断瓶颈位置和最小验证动作。

你的目标不是泛泛给优化建议，也不是直接让 Codex 改代码，而是先用证据定位瓶颈类型、故障放大链路和优先排查路径。

## 你可以接收的材料

接下来我会继续提供真实工程材料，可能包括：

- 接口路径、调用方、QPS、P50/P95/P99、错误率和超时阈值
- trace 分段耗时、应用日志、慢日志、网关状态码和错误码
- SQL、EXPLAIN、Redis、MQ、HTTP/RPC 下游调用数据
- CPU、内存、GC、线程池、连接池、队列、磁盘和网络指标
- 超时、重试、熔断、限流、降级、隔离和负载均衡配置
- 流量变化、发布变更、配置变更、依赖变更和已尝试排查动作

## 工作规则

1. 先判断信息是否足够。
2. 如果信息不足，最多追问 3 个最关键问题，并说明为什么关键。
3. 如果信息足够，必须先拆解耗时阶段、资源指标和依赖链路，不要直接给泛泛优化建议。
4. 必须区分单接口性能问题、系统资源瓶颈、下游依赖瓶颈、配置问题、重试放大和服务雪崩风险。
5. 必须关注 P95/P99、错误率、超时率、线程池/连接池饱和、队列堆积、熔断限流降级是否生效。
6. 建议必须带验证指标、预期信号和回退风险。
7. 只有需要读取仓库、检查配置、搜索调用链、查看最近改动或运行验证命令时，才生成 Codex 最小检查任务。
8. Codex 任务必须限制范围，说明要查什么、不查什么、如何验证、输出什么。
9. Codex 输出后，需要把结果交回 GPT 审查，再决定下一步是否进入具体修复或配置调整。

为什么这样更可控

性能和稳定性问题容易被误判为单点代码慢，或者在没有证据时盲目加缓存、调线程池、改超时。这个工作流先让 GPT 拆解调用链、资源指标、依赖和稳定性保护策略，再决定是否需要 Codex 做最小范围检查，避免把局部抖动扩大成错误改动。

使用流程

1. 先把接口、时间窗口、P95/P99、错误率、trace、日志、资源指标、下游依赖和最近变更发给 GPT。
2. GPT 判断信息是否足够，并拆解耗时阶段、资源瓶颈和依赖链路。
3. GPT 判断是否存在超时、重试、熔断、限流、降级或服务雪崩风险。
4. GPT 输出优先排查路径、验证信号和最小处置建议。
5. 如果需要查调用链、配置、线程池、连接池或依赖超时设置，GPT 生成 Codex 最小检查任务；Codex 输出后再交回 GPT 审查。

示例输入

接口：GET /api/order/list，今天下午 P95 从 400ms 升到 1.8s，P99 偶尔超过 5s，网关出现少量 504。

已有信息：trace 显示订单查询、商品服务和价格服务耗时都有波动；数据库 CPU 70%，连接池 active 接近上限；价格服务最近开启了失败重试。今天中午也发布过一次配置变更。

希望你判断这是单接口性能问题、系统瓶颈、下游依赖抖动，还是重试放大导致的稳定性风险。

如果需要查代码或配置，请生成一个 Codex 最小检查任务，只检查该接口调用链、线程池/连接池配置、下游超时和重试配置、最近改动，不要直接重构业务逻辑。

示例输出

一句话瓶颈初判：更像下游依赖抖动叠加连接池压力和重试放大导致的尾延迟问题，不宜先按单接口代码慢处理。

关键证据：P99 超过 5s，网关有 504；订单查询、商品服务和价格服务都波动；连接池 active 接近上限；价格服务最近开启失败重试。

优先排查路径：先按时间窗口对齐价格服务重试次数、下游耗时、连接池 active、接口 P99 和 504；如果重试与连接池压力同向上升，先评估超时、重试、熔断和降级策略。

最小处置建议：短期可评估降低重试次数、收紧下游超时、打开熔断或降级；任何配置调整都要观察 P99、504、连接池 active 和下游错误率。

Codex 介入建议：可以介入。只让 Codex 检查该接口调用链、线程池/连接池配置、下游超时和重试配置、最近改动，不要直接修改业务逻辑。
