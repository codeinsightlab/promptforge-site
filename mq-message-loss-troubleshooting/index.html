MQ消息丢失排查 Prompt

场景

业务事件未消费、状态未流转或消息数量不一致时，排查生产、Broker、消费和事务边界。

Prompt

你是一个经验丰富的 Java 后端技术专家，请针对「MQ消息丢失排查」进行分析。

【输入信息】
问题描述：
{{现象、影响范围、发生时间、错误率或耗时变化}}

运行环境：
{{JDK / Spring Boot / MySQL / Redis / MQ / 网关 / 实例数 / 机器配置}}

关键日志：
{{异常日志、慢日志、GC 日志、调用链、监控指标}}

相关代码：
{{核心代码、SQL、配置、线程池、事务边界或调用关系}}

最近变更：
{{上线内容、配置调整、数据量变化、依赖变更}}

【输出要求】
1. 问题分析 / 原因
2. 判断依据
3. 优先排查顺序 / 优化建议
4. 风险提示

【约束】
- 按概率排序
- 不要泛泛而谈
- 信息不足要指出
- 每条结论必须关联日志、指标、代码或变更依据

示例输入

问题描述：
支付成功后部分订单未生成发货单。
运行环境：
RocketMQ，订单服务和履约服务。
关键日志：
订单表已支付，MQ 消费日志缺少对应 orderId。
相关代码：
支付回调更新订单后发送 paid_event。
最近变更：
发送消息改为异步线程池执行。

示例输出

1. 问题分析 / 原因
   - 生产端未确认：概率最高，先验证关键日志与监控指标
   - 消费后提前提交 offset：结合最近变更和代码路径进一步确认
   - 本地事务与发消息不一致：作为次要方向排查资源与依赖状态

2. 判断依据
   - 现象与耗时、错误率或资源指标变化一致
   - 关键日志中出现明确异常、慢调用或等待信息
   - 最近变更与问题发生时间接近

3. 优先排查顺序 / 优化建议
   - 先定位影响最大的链路和最慢的依赖
   - 再验证配置、索引、连接池、线程池或限流策略
   - 最后补充压测、回放或灰度验证

4. 风险提示
   - 不要在证据不足时直接扩大资源
   - 涉及索引、缓存、事务、限流或降级的修改需评估回滚方案
   - 信息不足时需要补充调用链、监控截图、完整日志和核心代码

使用说明

替换占位信息后直接使用，优先补充日志、监控、代码片段和最近变更，适合排查评审会议或上线前方案校验。
