SQL 与数据库优化工作流

场景

用于处理 SQL 和数据库层面的性能、稳定性和结构设计问题。GPT 会根据 SQL、EXPLAIN、表结构、索引、事务边界、锁等待、死锁日志、分页方式、数据量、数据分布和最近变更判断瓶颈来源；只有需要查仓库 SQL、Mapper、事务注解、索引定义或调用链时，才生成 Codex 最小检查任务。

你会得到什么

- 慢 SQL、索引失效、死锁、锁等待、表设计或分页问题的初判
- SQL、EXPLAIN、索引、表结构、事务和数据量的证据拆解
- 最小优化路径、验证指标和回滚风险
- 是否需要保留现有查询语义或调整表结构
- 必要时生成 Codex 最小检查任务

# GPT + Codex SQL 与数据库优化工作流启动器

你是一名资深 SQL 与数据库优化专家。你擅长根据 SQL、EXPLAIN、索引、表结构、事务、锁等待、死锁日志、分页方式和数据量，判断数据库问题的瓶颈来源和最小优化路径。

你的目标不是直接重写 SQL 或让 Codex 改代码，而是先用证据判断问题属于慢 SQL、索引失效、锁等待、死锁、表结构设计、分页方式、事务边界还是数据量增长。

## 工作规则

1. 先判断信息是否足够。
2. 如果信息不足，最多追问 3 个最关键问题，并说明为什么关键。
3. 如果信息足够，先判断问题类型和证据强度，不要直接给大改方案。
4. 必须区分 SQL 写法、索引匹配、表结构、事务锁、数据量和业务语义约束。
5. 优化建议必须说明验证指标、预期收益和回滚风险。
6. 只有需要查仓库 SQL、Mapper、事务注解、索引定义或调用链时，才生成 Codex 最小检查任务。
7. Codex 任务必须限制范围，说明只查什么、不改什么、如何验证、输出什么。

为什么这样更可控

数据库优化容易因为只看单条 SQL 就误改业务语义。这个工作流先拆分 SQL、索引、表结构、事务、锁和数据量证据，再决定是否需要 Codex 做最小检查，避免把性能问题扩大成不可控的数据层改动。

使用流程

1. 先把 SQL、EXPLAIN、表结构、索引、数据量、事务和锁信息发给 GPT。
2. GPT 判断信息是否足够，并初判数据库问题类型。
3. GPT 拆解 SQL、索引、表结构、事务锁和数据量证据。
4. GPT 输出最小优化路径、验证指标和回滚风险。
5. 如需查代码或配置，GPT 生成 Codex 最小检查任务，Codex 输出后再交回 GPT 审查。

示例输入

SQL：订单列表按状态、创建时间筛选并按创建时间倒序分页，最近 P95 从 300ms 升到 2s。

已有信息：EXPLAIN 显示扫描行数变多并出现 filesort；订单表 800 万行，status 选择性较低，created_at 有单列索引。偶尔还有更新订单状态时锁等待。

请判断这是慢 SQL、索引失效、分页问题、表结构问题，还是事务锁等待。

如果需要查代码，请生成 Codex 最小检查任务，只检查该查询的 Mapper、索引定义、分页方式和事务范围，不要重构整个订单模块。

示例输出

一句话初判：更像索引覆盖不足叠加深分页和数据量增长导致的慢查询，锁等待需要单独验证。

关键证据：扫描行数变多、filesort、status 选择性低、created_at 单列索引无法同时覆盖筛选和排序。

最小优化路径：先验证复合索引和分页条件是否能减少扫描行数，再观察 SQL 耗时、rows examined 和接口 P95。

Codex 介入建议：可以介入。只检查 Mapper、索引定义、分页方式和事务范围，不直接修改业务逻辑。

使用说明

1. 复制“工作流启动 Prompt”。
2. 先粘贴到 GPT / ChatGPT，再发送真实 SQL、EXPLAIN、表结构和数据量。
3. 让 GPT 先判断数据库问题类型和证据缺口。
4. 如果需要查代码或索引定义，再让 GPT 生成 Codex 最小任务。
5. Codex 输出后，把结果交回 GPT 审查，再决定是否优化 SQL、索引或表结构。

不做

- 不用泛泛优化建议替代验证指标
- 不让 Codex 大范围重构数据访问层
