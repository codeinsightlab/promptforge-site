示例输出

## 1. 结论

- 是否建议立即优化：是
- 当前主要问题：现有索引只覆盖 tenant_id + status，无法同时满足 create_time 范围过滤和 ORDER BY create_time DESC，导致扫描行数较大并出现 filesort。
- 预期优化方向：新增组合索引，优先让查询按 tenant_id、status、create_time 过滤，并减少排序成本；同时关注 OFFSET 深分页问题。

## 2. 执行计划判断

- type 是否合理：range，基本可接受，但扫描范围过大。
- key 是否命中正确索引：当前命中 idx_tenant_status，不是最优索引。
- rows 是否过大：rows 约 180000，明显偏大。
- Extra 是否存在风险：
  - Using filesort：有风险，说明排序没有被索引完全承接。
  - Using temporary：当前未出现。
  - Using where：正常，但说明仍需 Server 层过滤。
  - Using index condition：有 ICP 优化，但不能解决扫描范围过大的问题。
  - Using join buffer：当前未出现。

## 3. 慢的主要原因

1. 问题：组合索引字段不足
   - 证据：当前 key 为 idx_tenant_status，Extra 出现 Using filesort
   - 影响：MySQL 需要先按 tenant_id/status 扫描大量数据，再按 create_time 排序

2. 问题：status 区分度较低
   - 证据：status 只有 6 种值，且 status IN (1,2,3)
   - 影响：单靠 tenant_id + status 不能有效缩小扫描范围

3. 问题：存在 OFFSET 深分页
   - 证据：LIMIT 20 OFFSET 5000
   - 影响：数据库需要跳过前 5000 条匹配记录，页码越靠后越慢

## 4. 索引优化建议

建议新增索引：

```sql
ALTER TABLE bus_order
ADD INDEX idx_tenant_status_create_time (tenant_id, status, create_time DESC);
```

字段顺序说明：
- tenant_id 放第一位：等值条件，且是多租户隔离字段，优先缩小范围。
- status 放第二位：等值 / IN 条件，继续缩小范围。
- create_time 放第三位：用于时间范围过滤，并尽量支持 ORDER BY create_time DESC。

是否覆盖 WHERE 条件：
- 可以覆盖 tenant_id、status、create_time 三个核心过滤字段。

是否支持 ORDER BY：
- 在 status 为单值时支持效果更好。
- 当前 status IN 多值时，仍可能存在额外排序，但扫描范围会明显小于原方案。

是否可能降低写入性能：
- 会。订单表新增索引会增加 insert/update 成本。
- 但该索引匹配高频后台查询，收益大于成本。

## 5. SQL改写建议

当前最小方案可以先不改 SQL，仅新增索引。

如果要进一步优化深分页，可改为游标分页：

```sql
SELECT
  o.id,
  o.order_no,
  o.user_id,
  o.status,
  o.pay_amount,
  o.create_time
FROM bus_order o
WHERE o.tenant_id = 10001
  AND o.status IN (1, 2, 3)
  AND o.create_time >= '2026-04-01 00:00:00'
  AND o.create_time <= '2026-04-30 23:59:59'
  AND o.create_time < '2026-04-20 12:30:00'
ORDER BY o.create_time DESC
LIMIT 20;
```

- 改写点：用上一页最后一条记录的 create_time 作为游标条件，减少 OFFSET 跳过成本。
- 预期收益：页码越深时收益越明显，避免扫描并丢弃大量已排序记录。
- 风险：需要前端或接口保存上一页游标，跳页能力会受限。

## 6. 分页优化建议

- 当前分页问题：OFFSET 5000 会导致数据库扫描并丢弃大量记录。
- 建议分页方式：优先改为基于 create_time + id 的游标分页，避免越翻越慢。
- 示例 SQL：

```sql
SELECT
  o.id,
  o.order_no,
  o.user_id,
  o.status,
  o.pay_amount,
  o.create_time
FROM bus_order o
WHERE o.tenant_id = 10001
  AND o.status IN (1, 2, 3)
  AND o.create_time >= '2026-04-01 00:00:00'
  AND o.create_time <= '2026-04-30 23:59:59'
  AND (
    o.create_time < '2026-04-20 12:30:00'
    OR (o.create_time = '2026-04-20 12:30:00' AND o.id < 8800123)
  )
ORDER BY o.create_time DESC, o.id DESC
LIMIT 20;
```

对应索引建议调整为：

```sql
ALTER TABLE bus_order
ADD INDEX idx_tenant_status_time_id (tenant_id, status, create_time DESC, id DESC);
```

## 7. 最小修改方案

- 改动内容：新增组合索引，先不改业务 SQL。
- 涉及 SQL：

```sql
ALTER TABLE bus_order
ADD INDEX idx_tenant_status_create_time (tenant_id, status, create_time DESC);
```

- 涉及索引：新增 idx_tenant_status_create_time
- 是否需要改代码：暂时不需要
- 风险等级：低到中

说明：
- 低风险点：不影响原查询语义。
- 中风险点：900 万行大表新增索引需要评估执行时间，建议低峰期执行。

## 8. 验证方式

- 需要重新执行的 EXPLAIN：
  - 确认 key 是否变为 idx_tenant_status_create_time
  - 确认 rows 是否明显下降
  - 观察 Extra 中 filesort 是否消失或影响降低
- 需要观察的指标：
  - SQL 平均耗时
  - rows examined
  - 慢查询日志出现频率
  - 后台订单列表接口 P95 响应时间
- 预期变化：
  - rows 从 180000 明显下降
  - 查询耗时从秒级下降到百毫秒级或数百毫秒级
  - 深分页场景仍可能较慢，需要后续游标分页继续优化
