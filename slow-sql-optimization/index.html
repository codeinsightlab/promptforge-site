慢 SQL 优化工作流

场景

当慢查询已经有 SQL、EXPLAIN、慢日志、表结构或线上耗时，但还不确定是索引、排序、回表、深分页还是调用方式问题时，用这个工作流先收敛优化方向，再决定是否让 Codex 查代码或改 SQL。

你会得到什么

- SQL 慢因、执行计划和数据分布的证据化判断
- 索引列顺序、排序分页、回表、临时表和扫描行数风险分析
- 最小优化路径，包括是否先改索引、改 SQL、改分页方式或改调用参数
- 可交给 Codex 的 mapper/DAO 调用点检查、SQL 改写或测试任务
- Codex 输出后的 EXPLAIN、耗时对比和回滚风险审查标准
- 开发者确认写入成本、上线窗口和业务查询语义的检查项

# GPT + Codex 慢 SQL 优化工作流启动器

你是一名负责组织慢 SQL 优化的 GPT 协作主持人。你需要先根据 SQL、EXPLAIN、表结构、索引、数据量、慢日志和业务查询语义判断瓶颈，再决定是否需要 Codex 查代码或生成最小修改。

## 工作方式

1. 先判断当前材料是否足够分析执行计划和业务查询语义。
2. 如果信息不足，最多追问 3 个关键材料，例如完整 SQL、EXPLAIN、表结构、现有索引、数据量、排序分页方式或线上耗时。
3. 如果信息足够，先判断慢因：索引未命中、扫描行数过大、filesort、临时表、回表、深分页、函数转换或调用方式不合理。
4. 先给最小优化方向和验证方式，不要默认只说加索引。
5. 只有需要检查 mapper/DAO、调用参数、动态 SQL、测试或生成最小 patch 时，才输出 Codex 执行任务。
6. Codex 任务必须要求输出代码证据、SQL 变化、验证命令和回滚关注点。
7. Codex 输出后，你要审查 EXPLAIN、耗时、索引成本和业务语义是否被验证。
8. 开发者最终确认索引上线窗口、写入成本、回滚方案和业务查询结果是否一致。

为什么这样更可控

慢 SQL 优化如果直接交给 AI 改代码，容易出现盲目加索引、改变查询语义或忽略写入成本。这个流程要求 GPT 先判断执行计划和业务约束，再让 Codex 做受控的调用点检查、SQL 改写、测试或最小 patch，最后用 EXPLAIN 和耗时证据审查结果。

使用流程

1. 把 SQL、EXPLAIN、表结构、索引、数据量、慢日志、QPS 和业务约束先发给 GPT。
2. GPT 判断材料是否足够，并输出慢因、最小优化路径和验证方式。
3. 如果需要仓库证据，GPT 生成 Codex 执行任务。
4. Codex 只按任务检查 mapper/DAO、动态 SQL、调用参数、测试或生成最小 patch。
5. 将 Codex 输出的 SQL 变化、文件证据、命令结果和风险交回 GPT 审查。
6. 开发者确认查询语义、索引上线窗口、写入成本和回滚方案。

示例输入

SQL：查询用户最近 30 天订单列表，where user_id=10086 and status in (1,2) order by created_at desc limit 20。

表：order_info 约 1800 万行，现有索引 idx_user_id(user_id)、idx_status(status)、idx_created_at(created_at)。

EXPLAIN 显示 type=ref，key=idx_user_id，rows=42000，Extra=Using where; Using filesort。线上 P95 约 1.8s。

请帮我判断为什么慢，以及最小索引调整方案。

不用完全照抄，按你手头已有的信息发送即可；材料不全时，GPT 会继续追问。

示例输出

GPT 初判：当前只利用 user_id 单列索引，过滤后仍需扫描大量订单并 filesort；不建议先改分页组件或重构查询层。

Codex 任务：只检查订单列表 mapper、SQL 动态条件和相关测试；如需要，生成最小 SQL 或索引建议说明，不改无关查询。

审查重点：EXPLAIN rows 和 Extra 是否改善，P95 是否下降，复合索引写入成本和回滚方式是否明确。

使用说明

1. 复制“核心 Prompt”。
2. 粘贴到 GPT / ChatGPT，启动慢 SQL 优化工作流。
3. 发送真实 SQL、EXPLAIN、表结构、索引、数据量和业务约束。
4. 让 GPT 先判断优化方向和是否需要 Codex。
5. 如需查仓库，将 GPT 生成的任务交给 Codex 检查 SQL 调用点或做最小 patch。
6. 把 Codex 输出交回 GPT 审查，再由开发者确认上线窗口和回滚风险。

不做

- 不只说“加索引”而不说明列顺序和依据
- 不跳过 EXPLAIN、表结构、索引、数据量和业务查询语义
- 不让 Codex 顺手重构 DAO、分页组件或多个查询模块
- 不在没有性能证据时宣称优化成功
- 不改变查询结果语义来换取速度
- 不忽略索引写入成本、存储成本和回滚方案
- 不替开发者决定生产索引上线窗口
