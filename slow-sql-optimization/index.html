慢SQL优化 Prompt

场景

根据 SQL、表结构、索引、数据量和执行计划，输出可验证的 MySQL 慢 SQL 优化建议。

Prompt

复制
你是一个 MySQL 与 Java 后端性能优化专家，请根据以下信息优化慢 SQL：

【输入信息】
SQL：
{{SQL}}

表结构：
{{表字段、主键、字段类型、现有索引}}

数据规模：
{{表数据量、过滤条件选择性、增长速度}}

执行情况：
{{执行耗时、扫描行数、返回行数、EXPLAIN、慢日志}}

业务约束：
{{是否允许改 SQL / 是否允许加索引 / 是否分页 / 一致性要求}}

【输出要求】
1. 慢的主要原因
2. 索引优化建议
3. SQL 改写建议
4. 风险提醒
5. 验证方式

【约束】
- 不要只给结论，必须说明判断依据
- 索引建议必须写出字段顺序
- 说明可能影响写入、锁表或存储成本的风险
- 如果执行计划不足，请指出需要补充的信息

示例输入

复制
SQL：
select * from order_info
where user_id = 10086
and status = 1
order by create_time desc
limit 20;

表结构：
order_info(id bigint PK, user_id bigint, status tinyint, amount decimal, create_time datetime, pay_time datetime)
现有索引：
idx_user_id(user_id)

数据规模：
order_info 约 8000 万行，单个高活跃用户最多 12 万订单。

执行情况：
耗时 1800ms，扫描 120000 行，返回 20 行。
EXPLAIN: key=idx_user_id, Extra=Using where; Using filesort

业务约束：
允许新增索引，接口只需要 id、amount、create_time、status 字段。

示例输出

复制
1. 慢的主要原因
   - 只命中 user_id 索引，status 过滤和 create_time 排序未充分利用索引
   - 扫描 120000 行但只返回 20 行，过滤和排序成本高
   - 执行计划出现 Using filesort

2. 索引优化建议
   - 新增 idx_user_status_create_time(user_id, status, create_time desc)
   - 如果 MySQL 版本不支持 desc 索引，也可使用同字段顺序索引验证排序效果

3. SQL 改写建议
   - 避免 select *，改为 select id, amount, create_time, status
   - 保持 where 条件与 order by 顺序匹配组合索引

4. 风险提醒
   - 新索引会增加订单写入成本和存储空间
   - 大表建索引需要评估线上 DDL 方式和执行窗口

5. 验证方式
   - 执行 EXPLAIN 确认 key 命中新索引
   - 观察 rows、filesort、接口 P95 是否下降
   - 用线上相似数据量回放查询验证

使用说明

粘贴 SQL、表结构、索引和 EXPLAIN，适合接口慢查询、列表分页和后台查询优化。
