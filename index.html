核心工作流

大多数真实开发任务，先从技术负责人工作流开始：判断需求边界、方案风险和是否需要 Codex 执行。

CORE WORKFLOW Tech Lead 技术负责人工作流 需求不清、方案不确定、担心过度设计或准备让 Codex 改代码前，先用它判断边界、风险和最小执行路径。 适合： 方案评估 · 复杂度控制 · Codex 任务拆分 · 结果审查 进入 Tech Lead →

常用开发场景

当问题类型已经比较明确时，直接进入对应场景工作流。

线上问题排查 线上异常、接口失败、日志报错或表现不一致时，先收敛影响范围、证据链和最可能根因。 启动排查 → 性能与稳定性瓶颈分析 接口变慢、超时、P95/P99 抖动或服务连锁异常时，拆解调用链、依赖、资源和配置瓶颈。 分析瓶颈 → SQL 与数据库优化 慢 SQL、索引失效、死锁或表结构不合理时，结合执行计划、事务和数据量判断优化路径。 优化数据库 → 缓存与 Redis 稳定性评审 评审 key、TTL、一致性、热点、大 Key、穿透、击穿和雪崩风险。 评审缓存 → 接口上线前评审 上线前检查接口幂等性、权限、安全、异常处理、兼容性和回归风险。 评审接口 → 分布式方案评审 评估分布式架构、数据一致性、事务边界、MQ 可靠性和补偿机制。 评审方案 → 发布与监控设计 设计灰度发布、回滚策略、监控指标、报警阈值和故障处理动作。 设计发布 →
