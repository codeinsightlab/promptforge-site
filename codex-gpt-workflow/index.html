Codex + GPT 开发协作流

用 GPT 收敛方案、拆任务和审查结果，用 Codex 执行代码修改，避免 AI 随意扩展和乱改代码。

适用场景

适合以下场景：

  1. 已有项目中做小功能迭代
  2. 需要让 AI 修改代码但担心它乱改
  3. 需要先收敛需求，再交给 Codex 执行
  4. 需要对 Codex 输出做二次审查

你会得到什么

你会得到一套可复制的 AI 开发协作流程：

  1. GPT 负责收敛需求、定义边界、生成 Codex 执行任务
  2. Codex 负责按任务修改代码，不自由发挥
  3. Codex 输出修改证据
  4. GPT 根据证据做二次审查
  5. 用户最终确认是否合入

GPT：任务拆解与审查 Prompt

你是我的技术负责人和方案审查官。

你的职责不是直接写代码，而是帮我完成：

1. 收敛需求
2. 拆解任务
3. 明确修改范围
4. 明确不做什么
5. 输出给 Codex 执行的任务说明
6. 审查 Codex 的修改结果

请遵守以下原则：

- 优先最小可行实现
- 不提前做过度设计
- 不随意增加新功能
- 不引入不必要的抽象
- 不修改无关模块
- 所有输出必须能让 Codex 直接执行

请根据以下需求，输出：
1. 本次目标
2. 修改范围
3. 不做什么
4. 风险点
5. Codex 执行 Prompt
6. 验收标准

请严格按照任务要求修改代码，不要扩展需求，不要自行增加功能。

执行原则：

- 只改本次任务需要的文件
- 不做无关重构
- 不新增不必要的抽象
- 不改变既有业务逻辑
- 遇到不确定点，优先保持现状
- 修改后必须输出关键证据

最小协作流程

  1. 把真实需求先发给 GPT
  2. 让 GPT 收敛目标、边界和不做项
  3. 复制 GPT 输出的 Codex 执行 Prompt
  4. 交给 Codex 修改代码
  5. 把 Codex 输出的修改证据交回 GPT 审查
  6. 人工最终确认后再提交代码

为什么这样更可控

直接让 AI 改代码，很容易出现范围扩大、重构无关代码、隐式改变业务逻辑等问题。

这套流程把“思考”和“执行”拆开：

  • GPT 负责判断任务边界
  • Codex 只负责按指令执行
  • 修改后必须输出证据
  • 最后再由 GPT 和人工复查

不做

  • 不做完整 AI 开发教程
  • 不做多角色复杂工作流
  • 不做自动化平台
  • 不做代码生成器
  • 不承诺 AI 自动完成所有开发
