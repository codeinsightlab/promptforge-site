Codex + GPT 开发协作流

用 GPT 收敛需求、设计方案和审查结果，用 Codex 执行代码修改、跑检查并输出证据，让开发任务从“讨论”稳定落到“可验收”。

场景

这是一个 AI 开发协作方法论页面，帮助开发者明确 GPT、Codex 和用户的职责边界。适用于 Java 后端功能开发、小程序前端修复、Bug 排查、接口改造、数据库字段调整和 PromptForge 页面批量优化。

GPT 职责

  • 收敛需求
  • 拆解任务
  • 冻结方案
  • 发现风险
  • 设计验收标准
  • 审查 Codex 输出

GPT 不应该直接假装已经改了代码。

Codex 职责

  • 阅读项目文件
  • 修改代码
  • 跑测试
  • 搜索引用
  • 输出 diff 证据
  • 验证是否只改了目标范围

Codex 不应该在需求没冻结时自行扩大范围。

用户职责

  • 提供真实业务背景
  • 确认取舍
  • 判断业务是否符合现场
  • 做最终验收

用户不需要一次性写完所有技术细节，但要确认边界。

协作流程 1~6

  1. GPT 先和用户讨论需求
  2. GPT 输出冻结方案
  3. GPT 生成 Codex 执行 Prompt
  4. Codex 修改代码并输出证据
  5. GPT 审查 Codex 输出
  6. 用户最终确认是否通过

## 2. 核心修改说明

## 3. 关键代码证据
```js
async contactMerchant() {
  const merchantId = this.data.details && this.data.details.merchantId
  if (!merchantId) {
    wx.showToast({ title: '商家信息加载中，请稍后重试', icon: 'none' })
    return
  }

  const imReady = await ensureIMReady()
  if (!imReady.ready || imReady.imUserId !== app.globalData.userInfo.id) {
    wx.showToast({ title: '聊天初始化中，请稍后重试', icon: 'none' })
    return
  }

  const conversationId = app.globalData.imChatManager.getConversationId(merchantId)
  wx.navigateTo({
    url: `/pages/contact/index?conversationId=${conversationId}`
  })
}
```

真实开发任务示例

先让 GPT 把“联系商家偶发进入错误会话”收敛为冻结需求：只改订单详情页点击前校验，不改 IM 模块整体设计，不改后端接口。再把冻结后的执行 Prompt 交给 Codex，让 Codex 搜索调用点、修改指定文件、运行检查并输出证据。最后由 GPT 根据文件清单、关键代码和测试结果审查是否越界。

为什么这样更可控

  • 需求和执行分离，避免边聊边乱改
  • GPT 先收敛边界，减少 Codex 误改范围
  • Codex 输出证据，方便 GPT 和用户审查
  • 每次只做一个明确目标，降低回滚成本

使用说明

先用 GPT 把需求冻结成“目标、业务规则、不做范围、允许修改、禁止修改、验收标准”。再复制本页 Prompt 交给 Codex 执行。Codex 完成后，把修改文件、关键证据和检查结果交回 GPT 或人工审查。
