死锁问题排查 Prompt

场景

数据库或 Java 锁竞争导致事务失败、线程阻塞时，分析死锁对象、锁顺序和修复方式。

Prompt

你是一个经验丰富的 Java 后端技术专家，请针对「死锁问题排查」进行分析。

【输入信息】
问题描述：
{{现象、影响范围、发生时间、错误率或耗时变化}}

运行环境：
{{JDK / Spring Boot / MySQL / Redis / MQ / 网关 / 实例数 / 机器配置}}

关键日志：
{{异常日志、慢日志、GC 日志、调用链、监控指标}}

相关代码：
{{核心代码、SQL、配置、线程池、事务边界或调用关系}}

最近变更：
{{上线内容、配置调整、数据量变化、依赖变更}}

【输出要求】
1. 问题分析 / 原因
2. 判断依据
3. 优先排查顺序 / 优化建议
4. 风险提示

【约束】
- 按概率排序
- 不要泛泛而谈
- 信息不足要指出
- 每条结论必须关联日志、指标、代码或变更依据

示例输入

问题描述：
并发修改订单状态时偶发 Deadlock found，失败率约 0.5%。
运行环境：
MySQL 8 InnoDB，隔离级别 RR，订单服务 6 实例。
关键日志：
show engine innodb status 显示 order_item 与 order_info 互相等待。
相关代码：
取消订单先更新 item 再更新 order，支付回调先更新 order 再更新 item。
最近变更：
新增批量取消订单任务。

示例输出

1. 问题分析 / 原因
   - 事务更新顺序不一致：概率最高，先验证关键日志与监控指标
   - 范围锁或间隙锁：结合最近变更和代码路径进一步确认
   - Java 锁嵌套顺序错误：作为次要方向排查资源与依赖状态

2. 判断依据
   - 现象与耗时、错误率或资源指标变化一致
   - 关键日志中出现明确异常、慢调用或等待信息
   - 最近变更与问题发生时间接近

3. 优先排查顺序 / 优化建议
   - 先定位影响最大的链路和最慢的依赖
   - 再验证配置、索引、连接池、线程池或限流策略
   - 最后补充压测、回放或灰度验证

4. 风险提示
   - 不要在证据不足时直接扩大资源
   - 涉及索引、缓存、事务、限流或降级的修改需评估回滚方案
   - 信息不足时需要补充调用链、监控截图、完整日志和核心代码

使用说明

替换占位信息后直接使用，优先补充日志、监控、代码片段和最近变更，适合排查评审会议或上线前方案校验。
