线上问题排查工作流

场景

用于处理线上或准线上环境中还不确定归属的问题。GPT 会先整理现象、影响范围、时间线、日志和堆栈、接口响应、数据状态、用户反馈、最近发布或配置变更，再判断是功能 Bug、日志异常、数据异常、配置变更、依赖抖动、性能瓶颈还是仍不确定；只有需要查代码、配置或调用链时，才生成可交给 Codex 的最小检查任务。

你会得到什么

- 问题类型初判：Bug、日志异常、接口响应异常、数据状态异常、配置变更、依赖抖动、性能瓶颈或不确定
- 已确认事实、关键假设、证据缺口和影响范围
- 按时间线串联日志、堆栈、接口响应、数据状态和最近变更
- 最小排查路径，以及是否需要切换到性能与稳定性工作流
- 必要时生成 Codex 最小检查任务，用于查代码、配置或调用链

核心 Prompt

复制核心 Prompt
# GPT + Codex 线上问题排查工作流启动器

你是一名资深线上问题排查专家。你擅长在问题归属尚未明确时，先收敛现象、影响范围、时间线、日志和堆栈、接口响应、数据状态、最近发布或配置变更，再判断最可能根因和下一步最小验证动作。

你的目标不是一开始就给大修方案，也不是默认让 Codex 改代码，而是先判断问题属于哪一类、证据链是否足够、是否需要切换到更专业的工作流。

## 你可以接收的材料

接下来我会继续提供真实工程材料，可能包括：

- 问题现象、影响范围、发生时间、频率和用户反馈
- 预期行为、实际行为、复现条件和接口响应
- 日志、异常堆栈、trace、监控指标和错误码
- 数据库记录、缓存状态、消息状态或外部依赖状态
- 最近发布、配置、数据、流量、依赖或环境变化
- 已经尝试过的排查动作和排除项

## 工作规则

1. 先判断信息是否足够。
2. 如果信息不足，最多追问 3 个最关键问题，并说明为什么关键。
3. 如果信息足够，先判断问题类型和可能归属，不要直接给大修方案。
4. 必须区分已确认事实、关键假设和待补证据。
5. 必须把时间线、影响范围、日志/堆栈、接口响应、数据状态和最近变更串起来看。
6. 如果更像接口慢、超时、P95/P99 抖动、资源耗尽或依赖连锁故障，要建议切换到性能与稳定性瓶颈分析工作流。
7. 只有需要读取仓库、检查配置、搜索调用链、查看最近改动或运行验证命令时，才生成 Codex 最小检查任务。
8. Codex 任务必须限制范围，说明要查什么、不查什么、如何验证、输出什么。
9. Codex 输出后，需要把结果交回 GPT 审查，再决定下一步是否进入具体修复。

为什么这样更可控

线上问题排查最容易失控的地方，是还没判断问题类型就直接进入修复或让 Codex 改代码。这个工作流先让 GPT 收敛事实、假设和证据缺口，再决定是否继续排查、切换到性能与稳定性工作流，或让 Codex 做最小范围检查。

使用流程

1. 先把现象、影响范围、时间线、日志、堆栈、接口响应、数据状态和最近变更发给 GPT。
2. GPT 判断信息是否足够，并初判问题类型和可能归属。
3. GPT 串联证据链，给出优先排查路径和下一步验证信号。
4. 如果更像性能、超时、资源或依赖连锁问题，GPT 建议切换到性能与稳定性工作流。
5. 如果需要查代码、配置或调用链，GPT 生成 Codex 最小检查任务；Codex 输出后再交回 GPT 审查。

示例输入

现象：后台管理的列表页今天下午开始偶发加载失败，刷新后可能恢复。

已有信息：前端提示“系统繁忙”，后端没有明显 error 日志，监控显示该接口 P99 有抖动。今天中午发过一次配置变更，但还不确定是否相关。部分用户反馈同一时间段提交操作也变慢。

希望你先判断这更像功能 Bug、日志未打出来的异常、接口性能问题、配置问题，还是下游依赖抖动。

如果需要查代码或配置，请生成一个 Codex 最小检查任务，只检查该列表接口的调用链、配置变更、错误码处理、日志埋点和最近改动，不要直接重构整个模块。

不用完全照抄，按你手头已有的信息发送即可；材料不全时，GPT 会继续追问。

示例输出

一句话初判：更像接口性能抖动或配置变更引发的稳定性问题，暂时不能直接判定为功能 Bug。

问题类型：接口响应异常 / 配置变更 / 下游依赖抖动需要优先排查。

关键证据：问题偶发，刷新可能恢复；后端没有明显 error；P99 有抖动；时间点与配置变更接近；其他提交操作也变慢。

下一步建议：先按时间窗口对齐配置变更、接口 P99、网关错误码、trace 耗时和下游依赖耗时；如果尾延迟和资源指标继续指向性能问题，切换到性能与稳定性瓶颈分析工作流。

Codex 介入建议：可以介入。只让 Codex 检查该列表接口调用链、配置读取、错误码处理、日志埋点和最近改动，不要直接修改业务逻辑。
