示例输入

复制 
问题描述：
生产环境订单提交接口偶发返回 500，用户重试后有时成功，高峰期更明显。

运行环境：
Spring Boot 2.7，MySQL 8.0，Redis 6，HikariCP，Linux，订单服务 4 个实例。

关键日志或现象：
java.sql.SQLTransientConnectionException: HikariPool-1 - Connection is not available, request timed out after 30000ms.
at com.example.order.OrderService.createOrder(OrderService.java:86)
接口 POST /api/orders 平均耗时从 180ms 升至 1200ms。

相关代码或配置：
spring.datasource.hikari.maximum-pool-size=20
新增 CouponService.checkAvailable(userId, couponId)，内部查询 user_coupon 表。

最近变更：
上午 10:00 上线优惠券校验逻辑，user_coupon 表数据约 6000 万。

示例输出

复制 
1. 数据库连接池耗尽（概率高）
   - 日志明确出现 Hikari 连接获取超时
   - 接口耗时上升与高峰期失败相符

2. 新增优惠券校验 SQL 变慢（概率高）
   - 最近变更与订单链路相关
   - user_coupon 数据量大，慢查询会长期占用连接

3. 高峰期并发超过连接池承载能力（概率中）
   - 偶发失败、重试成功符合资源竞争特征
   - 连接池最大 20，订单服务实例数有限

建议优先排查：
1. 查看 Hikari active、pending、timeout 指标
2. 抓取 CouponService.checkAvailable 的 SQL 与 EXPLAIN
3. 对比上线前后慢 SQL、QPS、数据库 CPU
4. 临时灰度关闭优惠券校验验证影响范围

需要补充：
- 失败时间段的慢 SQL 明细
- user_coupon 表索引
- 订单接口调用链耗时拆分

使用说明

把问题现象、日志、环境和最近变更替换到 Prompt 中，优先用于线上故障初步归因。
